# 项目管理业务流程

> 严格基于《解构领域驱动设计》20.3.2节（项目管理部分）

## 服务蓝图

当项目管理人员（通常为指定该项目的项目经理）开始发起项目立项的服务请求时，就形成了从立项到结项一个完整的项目管理流程。

### 关键角色定位

- **客户角色**：项目管理人员——项目管理人员扮演了服务蓝图的客户角色，整个流程的各个阶段都是为项目管理人员服务的
- **参与角色**：子公司、项目管理部、项目成员、服务中心——只参与项目管理流程，提供交互或支撑行为

> 如果要体现目标系统为项目成员提供的业务价值，就需要将项目成员当作服务蓝图的客户，思考它的客户旅程（如处理迭代问题的流程）。可以单独为这个流程绘制服务蓝图。由于视角不同，参与角色的身份也会发生变化。

## 业务场景与业务服务

### 业务场景1：项目管理

项目管理场景在流程中分为首尾两个阶段，在确定业务服务图时需要将这两个阶段的业务服务统一。

| 业务服务 | 说明 |
|---------|------|
| 创建项目 | 项目管理人员提交立项申请 |
| 审核项目立项 | 对立项申请进行审核 |
| 查看项目统计报表 | 查看项目相关的统计数据 |
| 查看项目进度 | 两个阶段的项目进度查看合并 |

### 业务场景2：资源分配

服务中心对硬件资源的分配支持了项目立项活动，为整个项目的项目组分配了资源，作为一种支撑活动放在一个单独的业务场景中。

| 业务服务 | 说明 |
|---------|------|
| 分配硬件资源 | 服务中心分配，支持项目立项活动 |

### 业务场景3：项目成员管理

| 业务服务 | 说明 |
|---------|------|
| 添加项目成员 | 将成员添加到项目团队 |
| 移除项目成员 | 从项目团队中移除成员 |

> 在添加或移除团队成员时，需要通过OA系统发送通知。通知的发送不在目标系统范围内，也不是由某个参与者发起，而是在添加或移除了团队成员之后进行，属于业务服务的执行步骤，不需要列入业务服务图。

### 业务场景4：项目计划管理

项目计划管理也分成了两个阶段，合并为一个业务服务图。

| 业务服务 | 说明 |
|---------|------|
| 创建迭代 | 创建项目计划的迭代周期 |
| 开始迭代 | 启动一个迭代 |
| 结束迭代 | 结束一个迭代 |

### 业务场景5：问题管理

| 业务服务 | 说明 |
|---------|------|
| 创建问题 | 创建问题并指定基本属性（标题、描述、问题类型等） |
| 分配问题给迭代 | 将问题分配到迭代周期中 |
| 分配问题给项目成员 | 将问题分配给项目成员使其成为问题的承担人 |
| 跟踪问题进度 | 跟踪问题在迭代周期内的过程 |

## 统一语言讨论：Issue vs Task

### 问题概念的确定

"问题"(Issue) 概念的获得并非一蹴而就。一开始倾向于使用"任务"(Task) 来表达这一概念。然而，在需求管理体系中，Task、UserStory、Epic、Defect 属于同一等级的概念，需要寻找到一个抽象概念来同时涵盖这几个概念，由此得到了"问题"(Issue) 概念。在 Jira 和 GitHub 的需求管理工具中，都使用了这一领域概念。

### 属性 vs 独立业务服务

项目管理者在创建问题时，会指定问题的基本属性（标题、描述、问题类型等）。那么，问题所属的迭代、承担人(owner)、报告人(reporter) 是否也属于问题的属性？

"分配问题给迭代"与"分配问题给项目成员"都可以认为是在编辑问题的属性。但既然业务服务为角色提供了服务价值，无论是将一个创建好的问题分配给迭代，还是将其分配给项目成员使其成为问题的承担人，都具有项目管理价值，是由项目管理者向目标系统发起的一次独立而完整的功能交互，应该分别识别为两个业务服务。

### 命名的修正

最初在确定项目管理的业务流程时，项目管理者要查看问题的完成情况以了解迭代进度，将流程中的一个活动命名为"查看问题完成情况"。在识别业务服务时，该名称没有清晰地体现该业务服务的服务价值。经过与业务分析人员沟通，认为该业务服务需要清晰地表达问题在迭代周期内的过程，准确的术语是"进度"(progress)，将其命名为"跟踪问题进度"(tracking issue progress) 更加符合该领域的统一语言。
