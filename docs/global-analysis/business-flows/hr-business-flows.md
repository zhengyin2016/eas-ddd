## 概述

本文档基于Day 3与HR总监王强的深度访谈，梳理出HR模块的5个核心业务流程，并以服务蓝图的形式进行描述。每个流程分为四个层次：顾客行为（前台触点）、前台服务、后台服务、支撑系统。

### 员工状态生命周期

在整个HR业务流程中，员工状态的变更是贯穿性的业务规则：

```
待入职 ──→ 试用期 ──→ 正式
              │          │
              │          ├──→ 调岗中 ──→ 正式（新岗位）
              │          │
              │          └──→ 离职中 ──→ 已离职
              │
              └──→ 离职中 ──→ 已离职
```

## 流程1：员工入职

### 基本信息

| 项目 | 内容 |
|------|------|
| 流程名称 | 员工入职 |
| 触发条件 | HR收到入职通知（新员工录用或内部调动） |
| 参与角色 | HR专员、新员工、部门经理、导师、系统管理员 |
| 平均耗时 | 3-5个工作日（从创建到完成入职培训） |

### 业务规则

1. 员工编号自动生成，规则为年份+四位序号（如 20260001）
2. 入职日期为实际报到日，非录用通知日
3. 每个员工必须关联一个部门和一个岗位
4. 系统账号在员工创建后自动生成
5. 导师由部门经理指定，通常为同部门资深员工

### 服务蓝图

```
顾客行为:    [HR收到入职通知] → [HR创建员工信息] → [员工填写个人信息] → [员工报到入职] → [员工参加培训]
                │                    │                    │                  │                │
前台服务:    [核实入职资料]    → [录入员工基本信息] → [员工自助完善信息] → [办理入职手续] → [入职培训签到]
                │                    │                    │                  │                │
后台服务:    [校验录用记录]    → [生成员工编号]     → [关联部门岗位]    → [创建系统账号] → [记录培训结果]
             [检查前置条件]      [设置初始状态]       [分配权限角色]      [发送欢迎通知]   [更新技能档案]
                                  [分配导师]                              [分配导师]
                                                                          [安排入职培训]
                │                    │                    │                  │                │
支撑系统:    [招聘系统]        [EAS-HR核心]         [组织架构模块]       [认证系统]       [培训管理模块]
             [OA审批流]         [编号生成器]          [岗位管理]          [权限系统]       [通知服务]
                                                                         [OA系统]
```

### 伴生系统交互

| 交互系统 | 交互内容 | 触发时机 |
|----------|----------|----------|
| OA系统 | 发送入职欢迎通知、通知部门经理 | 员工创建完成 |
| 认证系统 | 创建系统登录账号 | 员工创建完成 |
| 权限系统 | 设置初始角色和权限 | 关联岗位后 |
| 招聘系统 | 更新候选人状态为"已入职" | 入职确认后 |
| 培训系统 | 创建入职培训任务 | 入职手续完成后 |

## 流程2：员工离职/调岗

### 基本信息

| 项目 | 内容 |
|------|------|
| 流程名称 | 员工离职/调岗 |
| 触发条件 | 员工提交离职/调岗申请，或HR/部门经理发起 |
| 参与角色 | 员工、直属领导、HR专员、接收部门经理（调岗）、项目经理 |
| 平均耗时 | 离职30天、调岗7-15天 |

### 业务规则

1. 离职需完成所有项目交接，由项目经理确认
2. 调岗需目标部门经理确认接收
3. 离职申请需直属领导和HR两级审批
4. 调岗申请需直属领导、目标部门领导、HR三级审批
5. 离职后账号保留30天，之后冻结
6. 离职员工的项目分配自动释放

### 状态变更

- **离职路径**：在职 → 离职中 → 已离职
- **调岗路径**：在职 → 调岗中 → 新岗位在职

### 服务蓝图 — 离职

```
顾客行为:    [员工提交离职申请] → [直属领导审批] → [工作交接] → [HR办理手续] → [员工正式离职]
                  │                    │               │              │               │
前台服务:    [填写离职申请表]    → [审批/驳回]    → [交接清单确认] → [离职面谈]    → [离职确认签字]
             [选择离职原因]       [填写审批意见]    [项目交接确认]   [核算剩余假期]
                                                                     [核算薪资]
                  │                    │               │              │               │
后台服务:    [创建离职流程]      → [通知相关方]    → [检查项目状态] → [更新员工状态] → [归档员工记录]
             [检查进行中工作]      [通知HR]          [释放项目分配]   [冻结系统账号]   [生成离职证明]
                                 [通知项目经理]      [知识转移记录]   [回收资产]       [更新组织架构]
                  │                    │               │              │               │

```

### 服务蓝图 — 调岗

```
顾客行为:    [员工/经理发起调岗] → [双方领导审批] → [工作交接] → [到新部门报到]
                  │                     │               │              │
```

## 流程3：招聘管理

### 基本信息

| 项目 | 内容 |
|------|------|
| 流程名称 | 招聘管理 |
| 触发条件 | 用人部门提交招聘需求 |
| 参与角色 | 部门经理、HR招聘专员、面试官、候选人 |
| 平均耗时 | 30-60天（从需求到入职） |

### 业务规则

1. 每个招聘需求必须关联到具体的部门和岗位
2. 招聘需求审批至少需要部门经理和HR总监两级
3. 面试至少安排2轮：第一轮技术面试、第二轮综合面试
4. Offer发放需HR总监签字确认
5. 一个招聘需求可对应多个候选人

### 服务蓝图

```
顾客行为:    [部门经理提交需求] → [HR发布职位] → [候选人投递简历] → [候选人参加面试] → [候选人接受Offer]
                  │                  │                │                  │                 │
前台服务:    [填写招聘需求表]    → [编辑职位信息]  → [HR筛选简历]    → [安排面试日程]    → [发送Offer]
             [说明岗位要求]       [发布到招聘渠道]   [标记候选人评级]    [面试官评估打分]    [跟踪回复状态]
                                                                      [汇总面试反馈]
                  │                  │                │                  │                 │
后台服务:    [创建招聘需求]      → [审批需求]      → [解析简历信息]   → [记录面试结果]    → [生成Offer]
             [关联部门岗位]        [更新岗位编制]     [建立候选人档案]    [更新候选人状态]    [关联入职流程]
             [检查编制余量]        [同步招聘渠道]                        [触发下一轮或淘汰]
                  │                  │                │                  │                 │
支撑系统:    [组织架构模块]      [OA审批流]       [简历解析服务]     [面试管理系统]      [Offer管理系统]
             [编制管理]           [招聘平台API]     [候选人数据库]      [通知服务]          [入职系统]
```

### 招聘状态流转

```
需求阶段:    草稿 → 待审批 → 已批准 → 已发布 → 招聘中 → 已完成/已取消
候选人阶段:  待筛选 → 简历通过 → 面试中 → 面试通过 → Offer中 → 已接受 → 已入职
                                          → 面试未通过 → 已淘汰
```

## 流程4：考勤管理

### 基本信息

| 项目 | 内容 |
|------|------|
| 流程名称 | 考勤管理 |
| 触发条件 | 员工每日打卡 |
| 参与角色 | 员工、直属领导、HR专员 |
| 周期 | 每日记录，月度汇总 |

### 业务规则

1. 工作日需打卡两次（上班、下班），迟到/早退/缺勤自动标记
2. 迟到标准：上班打卡时间超过规定时间30分钟内为迟到，超过30分钟为严重迟到
3. 早退标准：下班打卡时间早于规定时间
4. 缺勤：未打卡且无请假记录
5. 请假需审批，至少直属领导一级审批
6. 加班需提前申请，由直属领导审批
7. 出差期间自动标记为"出差"，不计入考勤异常
8. 月度考勤汇总需在次月5日前完成

### 服务蓝图

```
顾客行为:    [员工上班打卡] → [员工下班打卡] → [员工提交请假/加班] → [领导审批] → [HR查看月度报表]
                  │                │                  │                  │              │
前台服务:    [移动端打卡]     → [移动端打卡]     → [填写申请表]       → [审批/驳回]    → [查看考勤汇总]
             [GPS定位记录]       [记录打卡时间]      [选择请假类型]        [填写意见]       [异常数据标注]
                                 [计算当日工时]      [上传附件]
                  │                │                  │                  │              │
后台服务:    [记录打卡数据]   → [计算日工时]      → [创建审批流]       → [更新请假记录]  → [生成月度报表]
             [校验打卡位置]      [标记迟到/早退]      [检查假期余额]        [更新考勤数据]    [统计异常天数]
             [校验打卡时间]      [关联排班计划]       [扣减假期余额]                         [核算薪资关联]
                  │                │                  │                  │              │
支撑系统:    [移动打卡服务]   [考勤计算引擎]       [OA审批流]          [通知服务]      [报表系统]
             [GPS服务]          [排班管理]          [假期余额管理]                       [薪资系统]
                                                                        [日历服务]
```

### 考勤异常处理流程

```
打卡异常 → 自动标记 → 通知员工 → 员工申诉/补卡 → 直属领导审批 → 更新考勤记录
                                                   → 审批驳回 → 维持异常标记
```

## 流程5：培训管理

### 基本信息

| 项目 | 内容 |
|------|------|
| 流程名称 | 培训管理 |
| 触发条件 | HR制定年度/季度培训计划，或部门提出培训需求 |
| 参与角色 | HR培训专员、部门经理、员工、培训讲师 |
| 周期 | 按计划执行，通常季度为单位 |

### 业务规则

1. 培训计划需关联技能标签，明确培训目标
2. 培训计划审批需HR总监确认
3. 员工可自主报名或由部门经理提名
4. 培训执行时需签到，签到率低于80%需重新安排
5. 培训完成后需收集反馈，评估培训效果
6. 完成培训后自动更新员工技能档案
7. 入职培训为必选项，不占用正常培训额度

### 服务蓝图

```
顾客行为:    [HR制定培训计划] → [员工报名/提名] → [员工参加培训] → [员工签到] → [员工填写反馈]
                  │                  │                │               │              │
前台服务:    [创建培训计划]     → [审批培训计划]   → [发送培训通知]  → [扫码签到]    → [反馈问卷]
             [关联技能标签]       [发布培训通知]      [培训资料下发]     [实时签到统计]    [培训评估]
             [设置培训名额]       [管理报名列表]
                                 [报名截止提醒]
                  │                  │                │               │              │
后台服务:    [关联技能体系]     → [检查报名资格]   → [准备培训资源]  → [记录出勤]    → [汇总反馈]
             [制定培训预算]        [名额控制]           [通知讲师]         [计算签到率]      [评估培训效果]
             [关联岗位能力模型]    [生成培训名单]                         [低于80%预警]     [更新技能档案]
                                                                                           [关联技能标签]
                  │                  │                │               │              │
支撑系统:    [技能管理模块]     [OA审批流]         [通知服务]       [签到系统]      [问卷系统]
             [预算管理系统]      [报名管理]          [资料管理系统]                    [技能档案]
             [岗位能力模型]                                                                 [报表系统]
```

### 培训状态流转

```
计划阶段:  草稿 → 待审批 → 已批准 → 已发布 → 报名中 → 报名截止 → 待执行 → 执行中 → 已完成
                                                                                          → 已取消
学员阶段:  待报名 → 已报名 → 已签到 → 培训中 → 已完成 → 已评估
                                     → 缺席 → 标记缺席
```

## 跨流程业务规则

以下业务规则贯穿多个流程，需要在建模时统一考虑：

| 编号 | 业务规则 | 涉及流程 |
|------|----------|----------|
| BR-001 | 员工状态变更遵循固定生命周期 | 入职、离职/调岗 |
| BR-002 | 每个员工必须关联一个部门和一个岗位 | 入职、调岗、组织架构 |
| BR-003 | 员工编号自动生成，规则为年份+四位序号 | 入职 |
| BR-004 | 所有审批流程至少两级（直属领导+HR） | 离职、调岗、招聘、请假、培训 |
| BR-005 | 培训签到率低于80%需重新安排 | 培训管理 |
| BR-006 | 离职需完成所有项目交接 | 离职 |
| BR-007 | 培训关联技能标签，完成后自动更新技能档案 | 培训管理 |
| BR-008 | 考勤异常自动标记，需员工申诉或补卡 | 考勤管理 |
| BR-009 | 面试至少2轮，第一轮技术、第二轮综合 | 招聘管理 |
| BR-010 | 系统账号生命周期与员工状态同步 | 入职、离职 |
