# EAS 系统愿景与范围

> 项目：企业资源管理系统（Enterprise Application System）
> 版本：v1.0
> 日期：Day 2

---

## 系统愿景声明

**构建一个以人力资源供需平衡为核心的企业数字化管理平台，通过统一的数据底座连接员工、项目和客户，实现集团资源的高效配置与运营透明化。**

---

## 当前状态（AS-IS）

### 组织背景

软件集团拥有300+名员工，分布在本部和多个子公司，业务涵盖软件开发、系统集成、IT服务等方向。组织结构为：集团决策层 -> 子公司管理层 -> 部门经理 -> 项目组。

### 现状痛点

| 问题域 | 现状描述 | 痛点 |
|--------|---------|------|
| 员工信息管理 | HR部门使用Excel维护员工信息表，300+人每人一行，30+字段 | 版本混乱、更新滞后、多人协作冲突、无法实时查询 |
| 招聘管理 | 全流程无系统支持，简历通过邮箱收集，进度靠口头沟通 | 流程不透明、候选人信息散落、无法统计招聘效率 |
| 考勤管理 | 使用钉钉打卡，但考勤数据与项目工时不关联 | 无法分析人员实际投入，工时统计靠手工 |
| 项目立项 | Word文档编写立项报告，审批通过邮件流转 | 格式不统一、审批状态难追踪、历史查询困难 |
| 人员分配 | PM之间通过口头或微信协调人员，无统一资源池视图 | 资源冲突频发、利用率不透明、抢人现象严重 |
| 进度跟踪 | 每周一开周会，Excel汇总进度，信息滞后一周 | 信息滞后、无法预警、问题发现太晚 |
| 问题管理 | 项目中的Issue通过邮件追踪 | 经常遗漏、无优先级排序、闭环率低 |
| 客户信息 | 每个销售用个人笔记本记录客户信息 | 信息不共享、离职带走、客户资产无法沉淀 |
| 商机管理 | 口头汇报销售进展，无法量化 | 销售漏斗不透明、预测不准确、管理靠感觉 |
| 合同管理 | 纸质合同归档在文件柜 | 查询困难、到期无提醒、电子版缺失 |
| 回款跟踪 | 财务手动对账，销售不知道回款状态 | 回款滞后、催收不及时、坏账风险 |

### 现状系统生态

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   钉钉        │     │  邮件系统     │     │  OA系统       │
│  (考勤打卡)   │     │  (审批沟通)   │     │  (行政审批)   │
└──────────────┘     └──────────────┘     └──────────────┘

┌──────────────┐     ┌──────────────┐
│  Excel        │     │  Word文档     │
│  (一切数据)   │     │  (一切报告)   │
└──────────────┘     └──────────────┘
```

**核心问题：没有统一的业务系统，数据散落在Excel、邮件、笔记本和口头沟通中。**

## 未来状态（TO-BE）

### 目标蓝图

EAS系统将建立统一的数字化管理平台，覆盖员工、项目和客户三大核心业务领域：

```
┌─────────────────────────────────────────────────────────┐
│                    EAS 统一管理平台                       │
│                                                         │
│  ┌───────────┐   ┌───────────┐   ┌───────────┐        │
│  │  HR 域     │   │  PM 域     │   │  CRM 域    │        │
│  │           │   │           │   │           │        │
│  │ ·员工管理  │   │ ·项目立项  │   │ ·客户管理  │        │
│  │ ·技能档案  │   │ ·资源分配  │   │ ·商机追踪  │        │
│  │ ·招聘流程  │   │ ·进度跟踪  │   │ ·合同管理  │        │
│  │ ·考勤管理  │   │ ·问题管理  │   │ ·回款跟踪  │        │
│  └───────────┘   └───────────┘   └───────────┘        │
│                                                         │
│  ┌───────────────────────────────────────────────┐     │
│  │           基础能力层                            │     │
│  │  组织架构 | 权限管理 | 消息通知 | 数据分析       │     │
│  └───────────────────────────────────────────────┘     │
│                                                         │
│  ┌───────────────────────────────────────────────┐     │
│  │           集成对接层                            │     │
│  │  OA系统 | 薪资系统 | 财务系统 | 钉钉            │     │
│  └───────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────┘
```

### 目标收益

| 维度 | AS-IS | TO-BE | 提升幅度 |
|------|-------|-------|---------|
| 人力资源利用率 | 不可量化 | 实时可见，目标提升15% | 15%+ |
| 进度信息滞后 | 1周 | 实时 | 从1周到实时 |
| 客户信息留存率 | 低（随人员流失） | 100%系统留存 | 质变 |
| 资源冲突发现 | 事后发现 | 事前预警 | 从被动到主动 |
| 数据查询效率 | 翻Excel/问人 | 系统秒查 | 量级提升 |

## 系统范围

### IN（包含在EAS系统范围内）

| 领域 | 功能范围 | 优先级 |
|------|---------|--------|
| **HR域** | 员工信息管理（增删改查、批量导入）、技能档案管理、组织架构同步、考勤数据同步与展示、招聘流程管理 | P0 |
| **PM域** | 项目立项与审批、项目团队成员分配、项目进度跟踪（里程碑+任务）、项目问题管理、资源池可视化 | P0 |
| **CRM域** | 客户信息管理、联系人管理、商机管理（销售漏斗）、合同管理（电子化归档）、回款跟踪 | P0 |
| **基础能力** | 组织架构管理、角色权限管理、消息通知、操作日志 | P0 |
| **数据分析** | 人力资源分布报表、项目进度汇总报表、销售漏斗报表 | P1 |
| **移动端** | 考勤打卡、个人信息查看、任务通知、审批处理 | P1 |

### OUT（不包含在EAS系统范围内）

| 功能 | 原因 | 处理方式 |
|------|------|---------|
| 薪资核算与发放 | 已有独立薪资管理系统 | 通过接口同步工时数据 |
| OA审批流程 | 已有OA系统 | EAS发起审批请求，OA执行流程 |
| 财务记账 | 属于财务系统职责 | 通过接口对接合同款项数据 |
| 代码管理 | 属于研发工具链 | 不纳入EAS范围 |
| 知识管理 | 属于独立需求，一期不纳入 | 二期考虑 |
| 即时通讯 | 钉钉已覆盖 | 不重复建设 |
| 项目文档协作 | 属于办公协作范畴 | 可后期集成 |

## 关键成功因素

| 编号 | 成功因素 | 说明 |
|------|---------|------|
| CSF-1 | **领域边界清晰** | HR、PM、CRM三大领域边界明确，领域间通过接口协作，避免耦合 |
| CSF-2 | **数据质量保障** | 历史数据清洗和迁移准确，新数据录入有校验机制 |
| CSF-3 | **用户接受度** | 系统操作体验优于Excel，用户愿意主动使用 |
| CSF-4 | **核心域识别准确** | 准确识别核心域并投入最优资源，确保核心竞争力的实现 |
| CSF-5 | **渐进式交付** | 按领域分批上线，快速验证，避免大爆炸式发布 |
| CSF-6 | **集成稳定性** | 与OA、薪资、财务等伴生系统的集成可靠稳定 |

## 约束条件

| 编号 | 约束类型 | 约束描述 | 影响 |
|------|---------|---------|------|
| CON-1 | 时间约束 | 一期上线时间6个月 | 影响功能优先级和交付策略 |
| CON-2 | 团队约束 | 核心开发团队5-8人 | 架构需适配团队规模，避免过度设计 |
| CON-3 | 技术约束 | 后端Java（Spring Boot），前端待定 | 技术选型受限，需在既有技术栈内选择 |
| CON-4 | 数据约束 | 历史数据在Excel中，格式不统一 | 数据迁移工作量较大，需要清洗和校验 |
| CON-5 | 集成约束 | OA系统接口能力有限 | 部分审批流程可能需要变通方案 |
| CON-6 | 用户约束 | 部分员工IT能力有限 | 移动端操作必须极简 |
| CON-7 | 安全约束 | 员工薪资等敏感信息不在EAS中 | HR域需明确数据边界，薪资相关字段不纳入 |

## 术语说明

| 术语 | 定义 |
|------|------|
| EAS | Enterprise Application System，企业应用系统 |
| 核心域 | 对企业核心竞争力有直接贡献的业务领域 |
| 支撑域 | 为核心域提供支撑能力的业务领域 |
| 通用域 | 跨领域通用的基础能力 |
| 子域 | 领域内部的细分业务范围 |
| 限界上下文 | 明确的领域边界，边界内的模型具有统一的含义 |
