## 1. 子领域划分策略

本系统采用**功能分类策略**（Function Classification Strategy）进行子领域划分。该策略基于业务功能对系统核心竞争力的贡献程度，将问题空间中的业务能力划分为三类子领域。

### 划分原则

- **业务价值导向**：以业务功能对"供需平衡"核心愿景的贡献度为判断标准
- **竞争力差异**：核心子领域是系统的差异化竞争力，通用子领域可替换，支撑子领域可外包
- **闭环完整性**：核心子领域组合必须形成完整的业务闭环
- **投资优先级**：核心子领域投入最多资源，通用子领域可采购成熟方案，支撑子领域可简略实现

## 2. 子领域类型定义

### 核心子领域（Core Subdomain）

核心子领域是系统的**核心竞争力所在**，承载了系统必须做到最好的业务能力。其特征：

- 直接贡献于系统的核心业务愿景
- 包含独特的业务规则和业务逻辑
- 是系统与竞品差异化的关键
- 需要持续投入最优资源进行设计和开发
- 失去核心子领域，系统将失去存在价值

### 通用子领域（Generic Subdomain）

通用子领域是**行业通用的基础能力**，几乎所有同类型系统都需要。其特征：

- 不具备行业特异性或竞争力差异
- 有成熟的行业标准或开源方案可参考
- 业务规则相对稳定和通用
- 可以采购或复用成熟解决方案
- 是系统运行的必要基础设施

### 支撑子领域（Supporting Subdomain）

支撑子领域是**辅助性的功能模块**，不创造核心业务价值，但为核心业务流程提供支持。其特征：

- 辅助核心业务流程的运转
- 不包含关键业务规则
- 实现相对简单，技术含量较低
- 可以外包或使用现成方案
- 缺失不影响核心业务目标的达成

## 4. 子领域详细说明

### 4.1 核心子领域

#### 人力资源管理（HR）

| 属性 | 说明 |
|------|------|
| 子领域类型 | 核心 |
| 核心愿景定位 | 供给方 — 管理企业的人力资源供给 |
| 战略重要性 | 人力资源是EAS"供需平衡"模型中的供给端，直接决定了企业能提供多少有效资源 |
| 包含的业务服务 | HR-01 ~ HR-12（共12个） |

**关键业务能力**：

- 员工全生命周期管理（入职、调岗、离职）
- 招聘流程管理（需求、面试）
- 员工能力管理（考勤、培训、技能档案）
- 储备人才库管理
- 人力资源供给查询

#### 项目过程管理（PM）

| 属性 | 说明 |
|------|------|
| 子领域类型 | 核心 |
| 核心愿景定位 | 需求方 — 管理项目需求和资源消耗 |
| 战略重要性 | 项目是EAS"供需平衡"模型中的需求端，项目的质量和效率是客户价值的核心体现 |
| 包含的业务服务 | PM-01 ~ PM-10（共10个） |

**关键业务能力**：

- 项目全生命周期管理（立项、执行、结项）
- 人力资源分配与调度
- 任务分解与进度跟踪
- 问题与风险管理
- 项目信息查询与分析

#### 客户关系管理（CRM）

| 属性 | 说明 |
|------|------|
| 子领域类型 | 核心 |
| 核心愿景定位 | 需求来源 — 管理客户需求和收入来源 |
| 战略重要性 | 客户是EAS"供需平衡"模型中的需求来源，没有准确的客户需求，供需匹配无从谈起 |
| 包含的业务服务 | CRM-01 ~ CRM-10（共10个） |

**关键业务能力**：

- 客户信息管理
- 销售商机全流程管理
- 合同与回款管理
- 销售数据分析

### 4.2 通用子领域

#### 组织结构管理

| 属性 | 说明 |
|------|------|
| 子领域类型 | 通用 |
| 说明 | 管理企业的部门、岗位、职级等组织架构信息 |
| 通用性理由 | 几乎所有企业应用都需要组织架构管理，无行业特异性 |
| 服务对象 | 为HR（员工分配组织）、PM（项目组组建）等核心子领域提供组织数据支撑 |

#### 认证授权

| 属性 | 说明 |
|------|------|
| 子领域类型 | 通用 |
| 说明 | 管理用户认证、角色权限和访问控制 |
| 通用性理由 | 标准的横切关注点，有成熟的行业方案（如OAuth2、RBAC） |
| 服务对象 | 为所有子领域提供身份认证和权限控制能力 |

### 4.3 支撑子领域

#### 消息通知

| 属性 | 说明 |
|------|------|
| 子领域类型 | 支撑 |
| 说明 | 管理系统内各类通知消息的发送和接收 |
| 支撑性理由 | 不创造核心业务价值，缺失不影响核心业务目标的达成 |
| 典型场景 | 入职通知、面试通知、项目状态变更通知、合同审批通知等 |

#### 文件管理

| 属性 | 说明 |
|------|------|
| 子领域类型 | 支撑 |
| 说明 | 管理系统中的文件上传、存储和下载 |
| 支撑性理由 | 通用辅助功能，可使用现成方案 |
| 典型场景 | 合同附件、项目文档、员工照片等 |

## 6. 核心子领域闭环关系

```
                    ┌─────────────┐
                    │  CRM 客户    │
                    │  关系管理     │
                    │  (需求来源)   │
                    └──────┬──────┘
                           │ 商机赢单触发项目
                           ▼
                    ┌─────────────┐
            ┌───────│  PM 项目     │───────┐
            │       │  过程管理     │       │
            │       │  (需求方)     │       │
            │       └─────────────┘       │
            │ 资源需求                    │ 资源分配
            ▼                              │
    ┌─────────────┐                        │
    │  HR 人力     │◄───────────────────────┘
    │  资源管理     │
    │  (供给方)     │
    └─────────────┘
           ▲
           │ 供需匹配
           └──────────────────────────┐
                                      │
            EAS 核心愿景：供需平衡 ◄────┘
```

三个核心子领域构成了EAS的核心业务闭环：

1. **CRM**（需求来源）：从客户需求中识别商机，赢单后触发项目创建
2. **PM**（需求方）：管理项目执行，向HR提出人力资源需求
3. **HR**（供给方）：管理人力资源供给，向PM提供可用人员

三者共同支撑了EAS的核心愿景——**供需平衡**。
