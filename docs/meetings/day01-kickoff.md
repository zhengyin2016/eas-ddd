## 一、项目背景介绍

张毅介绍了软件集团当前现状：

- **企业规模**：约300人，下设多个子公司
- **业务模式**：项目制管理，以项目为单位组织生产和交付
- **核心痛点**：
  - 人力资源调度完全依赖Excel共享表格，效率低下且容易出错
  - 人员技能信息分散，无法快速匹配合适资源
  - 客户管理混乱，销售离职导致客户信息断层
  - 项目进度缺乏统一可视化管理
  - 跨子公司资源协调困难

集团决策层已正式批准EAS系统建设立项。

## 二、项目目标讨论

### 2.1 核心愿景

张毅提出EAS系统的核心愿景：**实现人力资源的供需平衡**。

即：在项目需要人时能快速匹配到合适的人力资源，在项目间空闲期能合理调配人员，最大化人力资源利用率。

### 2.2 系统范围

EAS系统覆盖三大业务板块：

1. **人力资源管理（HRM）**：员工信息管理、技能档案、考勤、绩效评估、资源调度
2. **客户关系管理（CRM）**：客户信息管理、销售线索跟踪、商机管理、售后服务
3. **项目过程管理（PM）**：项目立项、任务分解、进度跟踪、资源分配、里程碑管理

### 2.3 项目约束

- 开发周期：30天
- 采用领域驱动设计方法论，严格按照《解构领域驱动设计》书中定义的流程执行
- 团队规模：9人

## 五、工作规范讨论

### 5.1 Git工作流

- 采用GitHub Flow：main分支为稳定分支，开发在feature分支进行
- 分支命名规范：`feature/{上下文名称}/{功能描述}`
- 提交信息格式：`[上下文] 动作: 描述`
- 合并需经过Code Review

### 5.2 统一语言

- 每个限界上下文维护自己的统一语言词汇表
- 代码中的命名必须与统一语言保持一致
- 统一语言变更需团队评审确认

### 5.3 每日Vlog

- 每日结束时由当日负责人撰写开发日志
- 记录当天工作内容、遇到的问题、关键决策和明日计划

## 七、技术选型讨论

周涛提出技术栈建议并获得团队一致通过：

| 技术组件 | 选型 | 选型理由 |
|---------|------|---------|
| 编程语言 | Java 17 | LTS版本，支持Record、Sealed Class等现代特性 |
| 应用框架 | Spring Boot 3.2 | 成熟稳定，生态完善，团队熟悉 |
| 持久层框架 | MyBatis | 对SQL控制更精细，聚合持久化策略更灵活 |
| 数据库 | MySQL 8 | 企业级成熟方案，团队运维经验丰富 |
| 数据库版本管理 | Flyway | SQL脚本版本化管理，支持团队协作 |

周涛说明了选择MyBatis而非JPA的理由：DDD中聚合的持久化策略复杂多样——一个聚合可能对应多张表、需要读写分离映射、需要精确控制懒加载边界。MyBatis对SQL的精细控制能力在这些场景下优于JPA的ORM自动映射。
