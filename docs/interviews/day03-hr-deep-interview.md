## 一、员工信息管理

### Q1：一个员工在系统里需要记录哪些基本信息？

**王强**：基础信息包括姓名、性别、出生日期、身份证号、手机号、邮箱、紧急联系人。工作相关信息包括员工编号、入职日期、部门、岗位、职级、直属领导、用工类型（正式/实习/外包）。还有一些是入职后逐步完善的，比如技能标签、项目经历、培训记录。

**李华**：员工编号的生成规则是什么？

**王强**：年份加四位序号，比如2026年入职的第一个人是20260001。每年重新从0001开始。这个编号是终身不变的，即使调岗或者离职后重新入职，编号也不变。

### Q2：员工有哪些状态？状态之间怎么流转？

**王强**：员工状态一共有六个：待入职、试用期、正式、调岗中、离职中、已离职。

流转规则是这样的：新员工创建后是"待入职"状态，实际报到后变为"试用期"。试用期通过后变为"正式"。如果员工申请调岗，审批通过后变为"调岗中"，到新部门报到后变回"正式"。如果员工申请离职，审批通过后变为"离职中"，完成所有交接后变为"已离职"。

有一点要注意：试用期员工如果离职，不需要"正式"这个中间状态，可以直接从"试用期"到"离职中"到"已离职"。所以状态流转不是严格的线性顺序。

**李华**：有没有可能从"已离职"回到其他状态？比如离职后重新入职？

**王强**：有可能，但不是简单的状态回退。我们会创建一条新的员工记录，沿用原来的员工编号，但状态从"待入职"重新开始。之前的工作记录会保留在历史档案里，不会丢失。

### Q3：员工的试用期是怎么管理的？

**王强**：试用期通常是三个月到六个月，根据合同类型和岗位级别不同。系统里需要记录试用期的起止日期，到期前两周HR会收到提醒。转正需要直属领导评估和审批。

试用期转正的评估内容包括工作表现、出勤情况、培训完成情况。如果评估不通过，可以延长试用期（最长不超过六个月）或者解除合同。不过实际操作中试用期不通过的案例很少。

### Q4：员工的技能档案包含哪些内容？

**王强**：技能档案包括三块。第一块是技能标签，就是员工具备哪些专业技能，比如Java、项目管理、需求分析之类的，每个技能有熟练程度评级——了解、熟悉、精通。第二块是项目经历，员工参与过哪些项目、在项目中担任什么角色、负责什么工作。第三块是证书，员工持有的专业认证，比如PMP、AWS认证这些，需要记录发证机构和有效期。

技能标签不是员工自己随便填的，有一个标准技能字典，员工只能从中选择。部门经理也可以给员工添加技能标签。培训完成之后，相关的技能标签会自动更新或者新增。

## 二、组织架构

### Q5：组织架构是怎么设计的？部门和岗位的关系是什么？

**王强**：组织架构分三层理解：公司、部门、岗位。公司下面有多个部门，每个部门有一个负责人。部门可以有自己的子部门，最多支持三级。比如技术部下面可以有前端组、后端组、测试组。

岗位是部门下面的，一个部门可以有多个岗位。比如技术部下面有Java开发工程师、前端开发工程师、测试工程师等岗位。每个岗位有对应的职级范围，比如Java开发工程师的职级范围是P4到P7。

**李华**：一个员工是不是只属于一个部门和一个岗位？

**王强**：对，在任意一个时间点，一个员工只能属于一个部门和担任一个岗位。但是历史记录要保留，比如员工从技术部的Java开发调到项目管理部的项目经理，这两段经历都要能查到。

### Q6：岗位和职级是怎么对应的？

**王强**：职级是P序列，从P1到P10，数字越大级别越高。每个岗位有一个职级范围。比如初级开发工程师是P3-P4，高级开发工程师是P5-P6，技术专家是P7-P9。员工在某个岗位上有一个具体的职级，晋升就是职级的提升，调岗可能伴随着职级的变化。

职级决定了几件事：薪资范围、审批权限、汇报层级。所以职级变更是需要审批的，不能随意修改。

### Q7：部门调整会影响已有员工吗？

**王强**：会影响。比如两个部门合并，原来属于这两个部门的所有员工都要调整到新部门。再比如一个部门拆分成两个，员工需要重新分配。这种组织架构调整是比较敏感的操作，一般由HR总监或者更高层级发起，需要详细的调整方案和审批。

系统需要支持批量调整，一个一个改太慢了。但是批量调整也要有审计记录，谁在什么时间做了什么调整，要能追溯。

## 三、招聘流程

### Q8：招聘流程从头到尾是怎样的？

**王强**：首先是用人部门提交招聘需求，说明要招什么岗位、多少人、什么要求、期望到岗时间。这个需求要经过部门经理和HR总监两级审批。审批通过后，HR发布招聘信息到各个渠道——招聘网站、猎头、内部推荐、校园招聘。

然后是收集简历。简历进来之后HR会做初筛，筛选通过的候选人进入面试环节。面试至少安排两轮：第一轮通常是技术面试或者专业面试，由用人部门的人来面；第二轮是综合面试，由部门经理和HR一起面。每轮面试之后面试官要写评估意见。

两轮面试都通过的候选人，HR会做背景调查和薪资谈判。都没问题的话发放Offer。候选人接受Offer后，进入入职流程。

### Q9：一个招聘需求可以对应多个候选人吗？

**王强**：当然可以。如果我要招3个Java开发，一个招聘需求编号下面可以有二三十个候选人。每个候选人独立走自己的流程——有的在简历筛选阶段，有的在面试阶段，有的已经发了Offer。最终根据Offer接受情况来关闭招聘需求。

招聘需求的状态有：草稿、待审批、已批准、已发布、招聘中、已完成、已取消。一旦招够了人，HR手动标记为"已完成"。如果业务需求变化了，可以取消招聘。

### Q10：面试评估有哪些维度？

**王强**：第一轮技术面试主要看专业技能、项目经验、解决问题的能力。面试官要给每个维度打分——优秀、良好、合格、不合格——然后写综合评价和推荐意见：强烈推荐、推荐、不推荐。

第二轮综合面试看的是沟通能力、团队协作、价值观匹配、职业规划。评分标准和第一轮一样。

两轮面试的意见要综合起来看。如果两轮都推荐，进入Offer环节。如果有一轮不推荐，需要HR和用人部门商量决定，不是直接淘汰。

### Q11：Offer发放之后候选人反悔怎么办？

**王强**：这种情况不少见。候选人接受Offer之后又反悔的，我们标记为"放弃入职"，然后从候选人池子里找备选。所以面试通过但还没发Offer的候选人，我们通常会保留一段时间，就是为了应对这种情况。

如果是发了Offer但候选人迟迟不回复，超过Offer上写的有效期就算自动放弃。

## 四、考勤管理

### Q12：考勤管理的基本规则是什么？

**王强**：工作日需要打卡两次——上班打卡和下班打卡。标准上班时间是9:00，下班时间是18:00，中午12:00到13:00是午休时间，不计入工时。所以标准日工时是8小时。

打卡用的是移动端APP，需要GPS定位，只有在公司范围内才能打卡，防止远程打卡作弊。

迟到和早退有区分：上班打卡超过9:00但在9:30之前，算迟到；超过9:30算严重迟到。下班打卡早于18:00算早退。没有打卡记录也没有请假记录的，算缺勤。

### Q13：请假和加班是怎么管理的？

**王强**：请假类型有年假、事假、病假、婚假、产假、丧假等。每种假期的审批规则不一样——年假和事假需要直属领导审批就行，病假需要提供医院证明，婚假和产假需要提供相关证明材料并且要HR审批。

请假申请需要提前提交，除非是突发情况。年假有额度限制，用完就不能再请了。系统要能实时显示每个员工的年假余额。

加班需要提前申请，直属领导审批通过才算有效加班。加班可以算调休或者加班费，员工在申请的时候选择。加班申请要关联到具体的项目或者工作任务。

### Q14：出差怎么算考勤？

**王强**：出差期间不需要打卡。员工提交出差申请，审批通过后，出差日期内的考勤自动标记为"出差"，不计入迟到早退统计。出差申请需要关联项目，说明出差的目的、时间、地点。

出差回来之后需要提交差旅报销，这个跟财务系统对接，不完全是HR的范畴。但是出差的记录要保留在员工的考勤记录里。

### Q15：月度考勤汇总是怎么做的？

**王强**：每月的考勤数据在次月5日前汇总。汇总内容包括：应出勤天数、实际出勤天数、迟到次数、早退次数、请假天数（按类型分）、加班时长、出差天数。

汇总结果需要HR审核确认。确认之后数据推送给财务系统，用于薪资计算。如果有异常数据——比如有员工某天既没打卡也没请假——HR需要逐条核实处理。

## 五、培训管理

### Q16：培训计划是怎么制定的？

**王强**：我们分年度计划和季度计划。年度计划是年初制定的，确定全年培训预算和大方向。季度计划是每个季度初细化的，确定具体的培训内容、时间、讲师、名额。

培训分几类：入职培训，每个新员工必须参加；岗位技能培训，针对特定岗位的专业技能；管理能力培训，针对管理岗位的领导力和管理技能；通用素养培训，比如沟通技巧、时间管理之类的。

每个培训计划要关联技能标签。也就是说，这个培训完成后，参训员工的哪些技能应该得到提升，这个要事先定义好。

### Q17：培训报名和执行是怎么管理的？

**王强**：培训计划审批通过后发布，员工可以看到并报名。有些培训是自由报名，先到先得；有些是指定参加，由部门经理提名。入职培训是强制的，新员工自动关联，不需要报名。

培训执行当天需要签到。我们用的是扫码签到，讲师在培训现场生成一个二维码，学员扫码确认到场。签到数据实时统计。

有一个硬性规则：如果培训签到率低于80%，说明培训组织有问题，需要重新安排时间和方式。这个规则是管理层定的，目的是确保培训不流于形式。

### Q18：培训效果怎么评估？

**王强**：培训结束后，学员需要填写反馈问卷，评价培训内容、讲师水平、培训组织。同时讲师也要对学员的表现做个评价。这些反馈汇总后，HR做培训效果评估。

更长远的效果评估是看员工的技能档案变化。比如一个Java高级开发的培训，三个月后看参训员工的技能标签是否有变化、项目表现是否有提升。不过这种长期评估实际操作比较难量化，目前主要还是看即时的反馈问卷。

## 六、绩效管理

### Q19：绩效管理是否在本次系统建设范围内？

**王强**：我建议这次先不做绩效管理。原因有几个：第一，绩效管理的业务规则很复杂，涉及到考核指标体系、评分标准、权重设置、申诉流程，每个部门可能都不一样。第二，我们公司目前的绩效考核方式还在调整，没有完全定型。第三，绩效数据的敏感性很高，权限控制和数据安全要求高。

建议放到二期。一期把员工管理、招聘、考勤、培训这些基础模块做好，二期再上绩效和薪酬。一步到位容易翻车。

**李华**：理解。那绩效相关的设计我们预留扩展点就行，对吧？

**王强**：对，技能档案可以算是绩效的基础数据，一期做好。后续绩效模块可以直接引用技能档案的数据。

## 七、HR与其他部门的协作

### Q20：HR和PM（项目管理）部门的人员分配是怎么协作的？

**王强**：PM部门在立项的时候需要从各个部门要人。流程是这样的：项目经理创建项目，确定需要哪些岗位的人、各需要多少人。然后向相关部门经理提交人员需求。部门经理根据部门人员情况决定派谁。确定之后，HR在系统里更新员工的项目分配信息。

一个员工在同一时间可以参与多个项目，但有一个主项目。项目结束后，员工的项目分配自动释放。如果员工离职或者调岗，所有项目分配都要清理，这个前面提过了。

对系统来说，关键是要实时反映每个员工的"可用状态"——在哪些项目上、每个项目分配了多少工时、还剩多少可用工时。这样PM部门要人的时候可以快速找到合适的人选。

### Q21：HR和财务的薪资对接是怎么做的？

**王强**：薪资核算的主体是财务部门，但很多基础数据来自HR——员工的职级、入职日期、考勤记录、加班记录、请假扣薪这些。

每月考勤汇总确认后，数据推送到财务系统。员工入职、调岗、离职的状态变更也要及时通知财务，因为涉及薪资调整、结算。

目前我们的做法是HR导出数据，财务导入自己的薪资系统。理想情况是系统之间直接对接，减少人工操作。

### Q22：有没有我遗漏的、但你觉得很重要的点？

**王强**：有两个。第一个是权限管理。不同角色看到的数据不一样。员工只能看自己的信息，部门经理能看本部门的，HR能看所有人的，但是薪资信息只有HR总监和财务能看。这个权限控制要细粒度的，按字段级别控制，不是简单的页面级控制。

第二个是审计日志。所有对员工数据的修改——谁改的、什么时候改的、改了什么——都要有记录。特别是状态变更、薪资相关字段的修改，这些是合规要求。

## 访谈总结

### 关键业务规则汇总

| 编号 | 业务规则 | 优先级 |
|------|----------|--------|
| HR-BR-01 | 员工编号规则：年份+四位序号，终身不变 | 高 |
| HR-BR-02 | 员工状态六态：待入职/试用期/正式/调岗中/离职中/已离职 | 高 |
| HR-BR-03 | 员工必须关联一个部门和一个岗位 | 高 |
| HR-BR-04 | 招聘需求审批至少两级（部门经理+HR总监） | 高 |
| HR-BR-05 | 面试至少2轮（技术面+综合面） | 高 |
| HR-BR-06 | 迟到标准：超过9:00为迟到，超过9:30为严重迟到 | 中 |
| HR-BR-07 | 考勤异常自动标记，需员工申诉或补卡 | 中 |
| HR-BR-08 | 培训签到率低于80%需重新安排 | 中 |
| HR-BR-09 | 培训完成后自动更新员工技能档案 | 中 |
| HR-BR-10 | 离职需完成所有项目交接 | 高 |
| HR-BR-11 | 调岗需目标部门经理确认接收 | 高 |
| HR-BR-12 | 技能标签来自标准技能字典，不可自定义 | 中 |
| HR-BR-13 | 所有数据变更需记录审计日志 | 高 |
| HR-BR-14 | 权限控制需到字段级别 | 高 |

### 一期不做的模块

- 绩效管理（业务规则未定型）
- 薪资核算（由财务系统负责）
- 猎头管理（外包流程）
