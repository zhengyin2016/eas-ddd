## Vlog：菱形对称架构

今天设计每个限界上下文的内部架构。

我在白板上画了一个菱形：

```
        ▲ 北向网关 (OHS)
       ╱ ╲
      ╱   ╲ 领域层
     ╱     ╲
    ╱       ╲
   ▼ 南向网关 (ACL)
```

这是**菱形对称架构**——我的书里详细讲过这个模式。核心思想是：
- 北向网关：对外提供开放主机服务(OHS)，定义稳定的API契约
- 南向网关：防腐层(ACL)，隔离外部依赖，保护领域模型
- 中间的领域层：纯粹的领域逻辑，不依赖外部

王强问：那数据库算哪一层？

好问题。数据库在南向网关之下，属于**基础设施**。领域层通过Repository接口访问数据，具体实现由基础设施层提供MyBatis Mapper。

赵敏问：那如果PM需要调用HR的员工服务，怎么调？

PM是下游，HR是上游。PM的南向网关会调用HR的北向网关。调用链是：

```
PM领域层 → PM南向网关(ACL) → HR北向网关(OHS) → HR领域层
```

这样PM的领域模型不会直接依赖HR的数据结构——如果有变化，只需要调整ACL，领域层不动。

## 技术选型

郑丽问我用什么技术栈。

我定了：

| 层级 | 技术选型 | 理由 |
|-----|---------|------|
| 应用框架 | Spring Boot 3.2 | 成熟稳定，团队熟悉 |
| ORM | MyBatis | SQL可控，性能优于JPA |
| 数据库 | MySQL 8.0 | 公司标准 |
| 迁移 | Flyway | 版本化数据库变更 |
| 消息队列 | RabbitMQ | 轻量级，满足需求 |
| API文档 | OpenAPI 3.0 | 标准化，可生成文档 |

孙磊问：为什么不用JPA？

我说：JPA是好的，但对于复杂查询，MyBatis的SQL可控性更好。而且这个项目有很多报表查询需求，JPA的Criteria API写起来很痛苦。
