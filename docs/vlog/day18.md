# Day 18 — 系统分层架构与代码模型

**日期**：架构映射 Day 18
**作者**：张毅

---

今天是架构映射阶段的最后一个技术环节：确定系统分层架构和代码模型。这一步将前面确定的12个限界上下文组装成一个完整的系统架构。

首先是系统分层架构。在系统上下文的约束下，根据限界上下文所属的子领域类型，将它们映射到不同的层次。订单、合同、员工、储备人才、招聘、项目、培训、决策分析这8个限界上下文属于核心子领域，映射到业务价值层。文件共享和OA集成属于支撑子领域，组织和认证属于通用子领域，它们映射到基础层。

EAS系统需要为集团决策者提供移动端应用程序，满足决策分析要求，也利于实时了解市场动态、人员动态和项目进度。针对市场部、人力资源部、项目管理部等职能部门，主要提供Web前端。因此有必要引入一个边缘层来应对不同UI前端的需求，即BFF for Web和BFF for Mobile。

系统分层架构属于架构的逻辑视图，并没有确定限界上下文的通信边界。例如，OA集成上下文与其他限界上下文并不在一个进程中，但系统分层架构并不需要体现这一点。

在代码模型层面，每个限界上下文都引入了菱形对称架构，但存在细微差异。决策分析上下文是弱化的菱形对称架构——它的领域逻辑主要为统计分析，受技术决策影响，通常可以直接针对数据进行操作，无须建立领域模型，内部只包含北向网关与南向网关。南向网关是一个薄薄的数据访问层，从数据库获得的统计分析数据会直接转换为消息契约模型。

OA集成上下文是一个由防腐层发展起来的限界上下文。它的领域层只包含了组装消息内容的领域模型，事件订阅者为北向网关的远程服务，事件处理器为北向网关的应用服务，事件发布者属于南向网关。

文件共享上下文的定义打破了惯有的设计方式。它实现为基础层的库而非服务——不需要跨进程通信，封装的领域逻辑包括上传文件与下载文件的领域行为，以及规定属于不同类别的文件存放在文件服务器的不同位置。

代码模型遵循菱形对称架构规定的标准结构：message目录存放发布语言PL，northbound目录包含开放主机服务OHS（含remote远程服务、local本地服务、appservice应用服务），domain目录按聚合划分命名空间，southbound目录包含防腐层ACL（含port端口和adapter适配器）。整个项目还共享ddd-core共享内核。

至此，架构映射阶段的所有技术工作已完成。明天将召开阶段评审会。
