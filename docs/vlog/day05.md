# Day 5 — 项目管理流程：Issue 还是 Task？

今天梳理项目管理流程。这个流程的特点是项目管理人员既是服务蓝图的客户角色，也是整个流程的受益者——所有阶段都是为项目管理人员服务的。子公司、项目管理部、项目成员和服务中心只参与流程，提供交互或支撑行为。

服务蓝图从项目管理人员提交立项申请开始，经历了一个项目的完整生命周期——从立项到结项。这里有一个重要的视角区分：如果要体现目标系统为项目成员提供的业务价值，就需要把项目成员当作客户角色重新绘制服务蓝图，比如处理迭代问题的流程。视角不同，参与角色的身份也会变化。

项目管理流程分成5个业务场景：

1. **项目管理**：创建项目、审核项目立项、查看项目统计报表、查看项目进度。注意，项目管理场景在流程首尾两个阶段都有活动，需要合并统一。

2. **资源分配**：分配硬件资源。这是服务中心对项目立项活动的支撑，分配资源给整个项目组。单独作为一个业务场景。

3. **项目成员管理**：添加项目成员、移除项目成员。OA系统发送通知属于业务服务的执行步骤，不列入业务服务图。

4. **项目计划管理**：创建迭代、开始迭代、结束迭代。和项目管理一样分成了两个阶段，需要合并。

5. **问题管理**：创建问题、分配问题给迭代、分配问题给项目成员、跟踪问题进度。

今天最值得记录的是关于统一语言的讨论。

**"问题"(Issue) 还是 "任务"(Task)？** 一开始我倾向于用 Task。但在需求管理体系中，Task、UserStory、Epic、Defect 属于同一等级的概念，需要找到一个抽象概念来涵盖它们——Jira 和 GitHub 都用了 Issue，我们也采用。

**"分配问题给迭代"和"分配问题给项目成员"是属性编辑还是独立业务服务？** 这两个操作都可以看作在编辑问题的属性。但业务服务的判断标准是"是否为角色提供了独立而完整的服务价值"。将问题分配给迭代、分配给项目成员使其成为承担人——这些都是项目管理者向系统发起的独立功能交互，具有项目管理价值。所以它们是两个独立的业务服务。

**"查看问题完成情况"→"跟踪问题进度"**。最初命名为"查看问题完成情况"，但这个名称没有清晰体现服务价值。和业务分析人员沟通后，认为应该表达问题在迭代周期内的过程，准确术语是"进度"(progress)。"跟踪问题进度"(tracking issue progress) 更符合统一语言。

统一语言不是一次成型的，是在识别业务服务的过程中不断打磨的。
