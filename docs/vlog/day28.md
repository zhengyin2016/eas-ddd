# Day 28 -- 远程服务的实现

**日期**：融合设计 Day 28
**作者**：张毅

---

今天是融合设计的第三个环节：远程服务的实现。远程服务是菱形对称架构北向网关的最外层，直接面对外部调用者，是系统与外界交互的入口。

培训上下文的远程服务采用REST风格实现，每个远程服务类对应一个资源（Resource）。`TicketResource`处理培训票相关的HTTP请求，`TrainingResource`处理培训管理相关的HTTP请求。它们使用Spring MVC的注解定义路由、HTTP方法和内容协商。

远程服务的设计遵循一个关键原则：薄。远程服务不应该包含任何业务逻辑，它的职责仅限于接收HTTP请求、将消息契约对象传递给应用服务、将应用服务返回的结果封装为HTTP响应。如果远程服务变得厚重，那说明职责分配出了问题，业务逻辑应该下沉到应用服务或领域服务中。

为了避免在每个远程服务方法中重复编写异常处理和响应封装的代码，我们引入了`Resources`辅助类。这个类定义在共享内核`ddd-core`中，提供了统一的`execute()`方法模板。`execute()`方法接受一个函数式接口作为参数，在执行过程中自动捕获不同层次的异常并返回对应的HTTP状态码：`DomainException`和`ApplicationException`返回400 Bad Request，未预期的`Exception`返回500 Internal Server Error。正常执行则返回200 OK和包装了响应数据的`ResponseMessage`对象。

异常类别与HTTP状态码的对应关系体现了领域驱动设计的分层思想。领域异常和应用异常都属于"业务错误"，客户端通过正确的请求参数可以避免，因此对应4xx状态码。系统异常属于"服务端故障"，客户端无法预见也无法处理，对应5xx状态码。这种映射让API的语义更加清晰，调用者可以根据状态码判断问题的性质。

`TicketResource`提供了培训票的核心操作端点：查询票的详情、提名候选人、确认或取消票。`TrainingResource`提供了培训管理的端点：创建培训、发起签到、确认缺勤名单。每个端点的请求体和响应体都使用`message`包中的契约对象，确保领域模型不直接暴露给外部。

远程服务的测试策略需要仔细权衡。单元测试的价值有限，因为远程服务本身几乎没有逻辑。更有价值的是契约测试或集成测试。契约测试验证API的契约是否符合约定——请求格式、响应格式、状态码。集成测试则验证远程服务与应用服务、领域服务、资源库的端到端协作。

今天完成了培训上下文全部远程服务的实现和测试。至此，菱形对称架构的四个层次——北向远程服务、北向应用服务、领域层、南向资源库适配器——全部到位。明天，我们将审视完整的代码模型。
