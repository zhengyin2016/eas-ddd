# Day 25 - 三个核心上下文建模完成

**日期**: 第25天
**参与者**: 张毅、郑丽、周涛、吴斌、孙磊
**主题**: 三个限界上下文全部完成领域建模

今天是领域建模阶段的最后一天。

早上，我召集所有人开会。"HR、PM、CRM三个核心上下文的领域建模都已经完成，我们来过一遍。"

周涛先汇报HR上下文："Employee、Talent、Recruitment、Attendance、Training五个聚合，基本业务规则都已实现。单元测试覆盖了核心场景。"

吴斌汇报PM上下文："Project、Issue、Sprint、Allocation四个聚合，项目状态流转、人员分配这些核心逻辑都测试通过了。"

最后是郑丽汇报CRM上下文："Customer、Opportunity、Contract、Payment四个聚合，昨天孙磊确认了所有业务规则，代码基本完成了。"

"CRM和PM的集成点考虑了吗？"我问。

"考虑了。"郑丽说，"合同签订后需要通知PM上下文创建关联项目。我在CRM里定义了PMClientPort接口，通过ACL适配器实现。但目前是模拟实现，真实调用等PM上下文稳定后再对接。"

"很好。"我点头，"那回款数据对接财务系统呢？"

"这个也用ACL实现。"郑丽说，"Payment记录确认后，通过财务系统客户端适配器发送数据。"

我在白板上画出了三个核心上下文的集成关系：

```
CRM上下文 --合同签订--> PM上下文
CRM上下文 --回款数据--> 财务系统（外部）
HR上下文 --人员分配--> PM上下文
```

"这些跨上下文的交互都是通过ACL实现的。"我强调，"限界上下文之间不能直接依赖领域模型，只能通过防腐层进行数据转换。"

下午，大家开始写集成测试用例。

"CRM到PM的集成怎么测？"吴斌问。

"可以写一个集成测试。"我说，"模拟合同签订，然后验证PM上下文是否收到通知并创建了项目。但目前两个上下文都在开发中，我们可以先用Mock测试。"

"单元测试呢？"钱伟问。

"单元测试每个聚合都应该有。"我回答，"Customer、Opportunity、Contract、Payment，每个聚合的核心行为都要测试覆盖。"

傍晚，我宣布："三个核心上下文的领域建模阶段全部完成！"

大家欢呼起来。

"接下来是什么？"周涛问。

"集成测试、系统测试、验收测试。"我回答，"钱伟会主导这个阶段。我们需要把各个上下文集成起来，验证端到端的业务流程。"

"比如一个完整的销售流程？"孙磊问。

"对。"我点头，"从创建客户开始，到创建商机、推进阶段、签订合同、创建项目、人员分配，最后回款。这个流程会跨三个上下文，需要验证数据流转是否正确。"

我合上笔记本，感觉一种成就感。三个核心上下文的领域建模是整个项目最核心的阶段，我们做到了。
