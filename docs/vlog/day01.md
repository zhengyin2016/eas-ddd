# Day 1 — 项目启动

今天是 EAS 项目正式启动的第一天。

我所在的软件集团决定开发一套企业自动化系统（EAS），目标是解决集团内部长期困扰的一个核心问题——人力资源的供需平衡。集团的业务就是软件开发，人是最核心的生产要素。但现实情况是，项目多的时候人不够用，项目少的时候又有人闲着，这种供需错配严重影响了交付效率。

上午，团队集结完毕，一共 9 个人。大家来自不同背景：有后端开发、前端开发、测试，也有对业务流程比较熟悉的同事。我把项目的核心愿景明确告诉了大家：**人力资源的供需平衡**。这不是一个模糊的口号，而是整个系统的北极星——所有功能模块的设计都要围绕这个目标展开，确保集团决策者能够清晰地看到人力资源的供给和需求状况，并据此做出调配。

下午的讨论主要集中在工作方法上。我向大家介绍了我们将要采用的领域驱动设计（DDD）方法论。按照我写《解构领域驱动设计》时的思路，全局分析阶段的第一步是价值需求分析，而价值需求分析要从商业模式画布开始。商业模式画布能够帮助我们从全局视角审视系统的商业价值、利益相关者和核心资源，避免一上来就陷入技术细节。

团队里有人问：为什么要从商业模式画布开始，而不是直接写需求文档？我的回答是：需求文档关注的是"做什么"，而商业模式画布关注的是"为什么做"和"为谁做"。如果我们不理解系统的商业本质，写出来的需求很可能偏离方向。EAS 不是一个通用的 OA 系统，它有明确的商业目标——通过人力资源的精细化管理来支撑软件交付业务。

今天确定了接下来的节奏：明天开始绘制商业模式画布，梳理客户细分、价值主张、核心资源等关键要素。商业模式画布完成后，再逐步推进到利益相关者分析、系统愿景和系统范围的定义。

最后分配了任务：每个人回去以后思考自己理解的利益相关者有哪些，明天带到讨论中来。商业模式画布的质量取决于我们对业务的理解深度，这个过程不能急。

明天，商业模式画布。
