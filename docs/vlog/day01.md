# Day 01 - 项目启动：九人集结

> 2026-05-24 | 张毅

---

今天是第一天。说实话，昨晚没怎么睡好。

不是因为紧张——好吧，也有一点紧张。主要是兴奋。我写《解构领域驱动设计》这本书的时候，花了大量篇幅讲方法论、讲模式、讲原则，但说实话，书里的案例都是简化过的。现在终于要拿一个真实的企业级项目来验证了，成功了那是理所当然，失败了……那就是公开打脸。

九点钟，人到齐了。我扫了一眼会议室（好吧，是腾讯会议），九个人齐刷刷在线。

**团队阵容**：李华是业务分析师，我点名要的，他对企业级系统的业务嗅觉很敏锐；王强是HR领域专家，在软件集团人力资源部干了八年；赵敏是项目管理部的老兵，PMP证书比我的书还厚；孙磊负责销售领域，之前在CRM厂商做过三年产品；三个后端——周涛管HR上下文，吴斌管PM上下文，郑丽管CRM上下文；还有钱伟，我们的测试工程师，据说手特别黑，专挑边界条件。

我先简单介绍了项目背景。软件集团，300人规模，下面有好几个子公司，做项目制的业务。现在的问题是什么？人力资源调度全靠Excel——你没听错，300人的公司，排兵布阵用的是共享表格。客户管理更混乱，销售走了客户信息就断了。集团决策层忍了很久，终于决定上一个EAS系统。

EAS的核心愿景，我说得很直接：**实现人力资源的供需平衡**。说白了就是——项目要人的时候有合适的人上，项目不缺人的时候人别闲着。听起来简单，做起来就知道有多难。

系统范围定了三大板块：人力资源管理、客户关系管理、项目过程管理。我特意跟团队强调了，这三个不是三个模块，而是三个问题域，后面要做限界上下文拆分的，现在别急着写代码。

下午讨论技术选型。我本来倾向于Spring Boot + JPA，毕竟JPA的聚合根持久化天然契合DDD。但周涛提了一个很好的观点：MyBatis对SQL的控制更精细，聚合的持久化策略可以更灵活——比如一个聚合对应多张表、读写分离的映射，MyBatis都能精确控制，JPA的ORM映射反而有时候会成为约束。我仔细想了一下，觉得他说得有道理。团队讨论后一致通过：**Java 17 + Spring Boot 3.2 + MyBatis + MySQL 8 + Flyway做数据库版本管理**。

技术栈定了，周涛和吴斌就去初始化GitHub仓库和Maven多模块项目骨架了。我看了一眼他们搭的结构，还算清晰，按上下文分模块，聚合层、领域层、基础设施层都有预留。第一天能把这些搭起来，效率不错。

我们还一起过了一遍利益相关者名单，识别了8类：集团决策者、市场部、人力资源部、项目管理部、子公司管理层、服务中心、财务部、普通员工。后面做需求分析的时候要逐个访谈。

最后确认了30天的执行计划，分四个阶段：全局分析7天、架构映射7天、领域建模11天、集成测试5天。时间很紧，但没有退路。

**明天计划**：李华带着大家画商业模式画布，开始做价值需求分析。这是整个项目的起点——如果价值都定义不清楚，后面的建模全是空中楼阁。

散会的时候快七点了。我关掉会议窗口，看着屏幕上还没关掉的PPT，心想：这就是开始了。三十天后见分晓。

有点兴奋，也有点压力。但更多的是一种踏实感——方法论是现成的，团队是靠谱的，剩下的就是干。

Day 1，完成。
