# Day 6 — 业务场景与业务服务识别

今天的工作是把前三天梳理出来的业务流程，按照"业务目标"切分为业务场景，再从业务场景中提炼出业务服务。这一步在《解构领域驱动设计》的全局分析阶段里至关重要——如果说业务流程是"线"，那业务场景就是线上的"节点"，业务服务则是每个节点背后的"动作"。

我用了一个自己总结的方法：**每个业务流程的"前台触点"本质上就是一个业务场景，"后台流程"中的每一步本质上就是一个业务服务。** 这个方法的好处是让识别过程变得可操作——不是凭空想象"应该有哪些服务"，而是从已验证的业务流程中逐层拆解。

以HR领域的"员工入职"流程为例。入职流程的前台触点是HR在系统中操作"创建新员工"的那一刻——这就是一个业务场景，我把它命名为"HR创建员工"。但这个场景背后有一系列后台动作：创建员工记录、分配员工编号、关联组织（部门和岗位）、创建系统账号。这些每一步都是一个独立的业务服务。为什么要拆到这么细？因为在后续的架构映射阶段，不同的业务服务可能归属不同的限界上下文，拆得越清晰，映射就越准确。

上午带着团队一起做HR领域的拆分。李华上手很快，把招聘流程和考勤流程拆得井井有条。但在拆员工入职流程时，他犯了一个典型的错误：把"发送入职通知邮件"也列为核心业务服务。我当场纠正了他。

我说，通知是一个支撑功能，不是核心业务服务。核心业务服务必须直接贡献于业务价值——"创建员工记录"直接支撑了企业的人力资源管理能力，"分配组织"直接决定了组织架构的完整性，而"发送入职通知"只是在完成核心操作之后的一个辅助动作。区分核心和支撑的判断标准是：**如果去掉这个服务，业务目标是否还能达成？** 去掉"创建员工记录"，入职流程就无法进行；去掉"发送入职通知"，员工只是少收到一封邮件，入职本身不受影响。

李华理解了，但紧接着又问了一个好问题："那如果是入职当天必须签署的电子合同呢？这个算不算核心业务服务？"这个问题的边界确实模糊。我最后把它归为"关联组织"业务服务的一部分，因为电子合同签署本质上是确认员工与组织的法律关系。

下午继续拆分PM领域和CRM领域。PM领域的拆分相对顺利——项目立项、人员分配、任务管理、进度跟踪，每个流程都对应了清晰的前台触点和后台动作。CRM领域稍微复杂一些，因为"商机管理"涉及多个阶段的流转，每个阶段都有不同的业务服务。我最终把它拆为"创建商机"和"推进商机阶段"两个核心服务，而不是按销售漏斗的每个阶段各建一个服务——那样粒度太细，会让服务数量膨胀。

到傍晚，三个领域加起来一共识别出约32个业务服务。HR领域12个，PM领域10个，CRM领域10个。我把它们全部整理成了业务服务识别文档，每个服务都有对应的业务场景、简要描述和所属领域。这个文档将成为明天子领域划分的重要输入。

今天的核心收获是：**业务服务不是被"发明"的，而是从业务流程中"发现"的。** 只要你有一个清晰的拆分方法，并且严格区分核心与支撑，识别过程就不会失控。

明天要做两件事：一是用功能分类策略将问题空间划分为核心、通用和支撑子领域；二是召开全局分析阶段评审会，邀请所有领域专家确认我们的成果。全局分析阶段即将收尾，我既期待又有些紧张——评审会上总会有人提出你没考虑到的东西。
