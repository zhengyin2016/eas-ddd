## Vlog：C4容器图

今天画系统分层架构——C4 Level 2。

我问郑丽：要不要加API Gateway？

郑丽有点犹豫，说好像大家都用API Gateway。

我说：**30天项目，能省就省**。

API Gateway的好处是统一的认证、限流、路由。但我们的场景：
- 三个后端服务直接暴露给前端，也没问题
- 认证用JWT，前端自己带token就行
- 限流暂时不需要

所以我的决策是：**直接用Spring Boot的DispatcherServlet**，以后真需要了再加Gateway。

## 系统分层

我们定了三层架构：

```
┌─────────────────────────────────────────────────────────────┐
│                    边缘层 (Edge Layer)                       │
│                  Web前端 + Mobile前端                        │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼ HTTP/WebSocket
┌─────────────────────────────────────────────────────────────┐
│                  业务价值层 (Business Layer)                 │
│     ┌───────────┐ ┌───────────┐ ┌───────────────────┐      │
│     │ HR上下文  │ │ PM上下文  │ │ CRM上下文          │      │
│     │ (8080)    │ │ (8081)    │ │ (8082)             │      │
│     └───────────┘ └───────────┘ └───────────────────┘      │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼ REST / 消息队列
┌─────────────────────────────────────────────────────────────┐
│                    基础层 (Foundation Layer)                 │
│     ┌───────────┐ ┌───────────┐ ┌───────────────────┐      │
│     │ 组织上下文 │ │ 认证上下文 │ │ 通知上下文         │      │
│     │ (8083)     │ │ (8084)     │ │ (8085)             │      │
│     └───────────┘ └───────────┘ └───────────────────┘      │
│                                                             │
│     ┌─────────────────────────────────────────────────┐    │
│     │              共享内核 (eas-common)               │    │
│     │  • 领域事件契约  • 公共DTO  • 工具类            │    │
│     └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                   基础设施层 (Infrastructure)                │
│  MySQL | RabbitMQ | Redis | Nginx | Docker                  │
└─────────────────────────────────────────────────────────────┘
```

## 端口分配

郑丽问端口怎么分配。

我说：

| 服务 | 端口 | 说明 |
|-----|------|------|
| eas-hr-context | 8080 | HR上下文 |
| eas-pm-context | 8081 | PM上下文 |
| eas-crm-context | 8082 | CRM上下文 |
| eas-org-context | 8083 | 组织上下文 |
| eas-auth-context | 8084 | 认证上下文 |
| eas-notification-context | 8085 | 通知上下文 |

前端通过Nginx反向代理访问这些端口。

## 部署架构

孙磊问：每个上下文独立部署？

对，这就是**微服务**的好处——每个上下文可以独立部署、独立扩展。

但孙磊又问：30天项目，用Docker还是直接jar包？

我说：**Docker**。原因：
1. 环境隔离，开发/测试/生产一致
2. 以后上K也方便
3. Docker Compose一键启动全部服务
