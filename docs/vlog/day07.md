# Day 7 — 子领域划分与全局分析阶段评审

今天是全局分析阶段的最后一天，主要做两件事：子领域划分，以及全局分析阶段评审会。

上午开始子领域划分。我用了《解构领域驱动设计》中提到的"功能分类策略"，将EAS的问题空间划分为核心子领域、通用子领域和支撑子领域三类。核心子领域是系统的核心竞争力所在，通用子领域是行业通用的基础能力，支撑子领域则是辅助性的功能模块。

划分过程远比我预想的要激烈。

我先提出了一个划分方案：把"人力资源管理"归为核心子领域。理由很充分——EAS的核心愿景是"供需平衡"，企业的人力资源是"供给方"，没有供给，平衡无从谈起。HR管理的质量直接决定了企业能提供多少有效的人力资源，这当然是核心。

赵敏立刻反驳。她说"项目管理"才是核心子领域，因为项目是"需求方"。没有项目需求，再好的供给也没有意义。企业的存在是为了交付项目，项目的质量和效率才是客户关心的，客户不会因为你员工管理得好就买单。

会议室里一度安静了几秒。我承认赵敏说得也有道理。

然后我突然意识到一个问题：我们争论的焦点是"谁是核心"，但真正的核心既不是HR也不是PM——**"供需匹配"本身才是EAS的核心竞争力。** HR是供给方，PM是需求方，CRM是客户方（需求来源），三者共同构成了一个闭环。把这个闭环中的任何一环拿掉，"供需平衡"这个核心愿景就实现不了。

所以我做了一个决定：把HR和PM都归为核心子领域。CRM也不例外——客户关系管理直接决定了需求的质量和来源，没有准确的客户需求信息，PM就无法有效管理项目。三个领域共同构成核心，缺一不可。

赵敏对这个结论表示满意。李华也点头同意。我想，这个争论本身就是有价值的——它让我们更清楚地理解了EAS的真正核心。

最终的子领域划分方案如下：

**核心子领域**包括人力资源管理、项目过程管理和客户关系管理。这三个领域直接承载了EAS"供需平衡"的核心愿景，是系统必须做到最好的部分。

**通用子领域**包括组织结构管理和认证授权。这两个功能几乎所有企业应用都需要，不具备行业特异性，但又是系统运行的基础设施。

**支撑子领域**包括消息通知和文件管理。它们辅助核心业务流程的运转，但本身不创造核心业务价值。

下午召开了全局分析阶段评审会。我邀请了所有参与过领域访谈的领域专家——王强、赵敏、李华、陈总以及几位业务骨干。会议的目的是让所有利益相关者确认我们的分析成果，确保没有遗漏关键的业务场景。

评审会按照领域逐一过业务流程和业务服务。HR领域和PM领域的评审比较顺利，大家对识别出的业务服务基本没有异议。但在评审到HR领域时，王强提出了一个重要遗漏：**储备人才管理。**

他说，除了已入职的员工，公司还有一个"人才库"——那些还没入职但已经通过面试、有明确意向的候选人。这些人的信息需要管理起来，当有新项目启动时可以快速联系和入职。当前的HR业务流程完全覆盖了"已入职员工"的生命周期，但对"待入职人才"的管理是一片空白。

我当场承认这是一个遗漏。储备人才管理确实是HR模块需要覆盖的业务场景，而且在"供需平衡"的框架下，储备人才是"潜在供给"，其重要性不言而喻。我把它补充进了HR领域的业务服务列表中，命名为"管理储备人才"。

评审会最后，我做了全局分析阶段的总结。过去五天（Day 3-7），我们完成了五项工作：理解系统愿景、识别业务流程、统一通用语言、识别业务服务、划分子领域。这些成果将以文档形式归档，作为后续架构映射阶段的基础。

评审决议：**全局分析阶段通过，进入架构映射阶段。**

散会后我一个人坐在会议室里，看着白板上满满的业务服务列表和子领域映射图，心里有一种踏实的满足感。从Day 1的客户沟通到今天的评审通过，我们用一周时间完成了对EAS问题空间的完整剖析。虽然过程中有争论、有遗漏、有纠正，但最终的成果是经得起推敲的。

明天开始进入架构映射阶段。根据《解构领域驱动设计》的方法论，我们要做的事情是：确定限界上下文、绘制上下文映射、定义上下文之间的关系。这是从"问题空间"到"解决方案空间"的桥梁——也是DDD实践中最具挑战性的环节之一。

我准备好了。
