# Day 7 — 三大业务流程梳理完成

连续四天的业务流程梳理工作终于完成。回顾一下成果：

**客户合作业务流程**——4个业务场景，13个业务服务。最重要的收获是建立了"订单"概念，厘清了合同、订单、市场需求和客户需求之间的关系。这个统一语言的建立过程比识别业务服务本身更有价值。

**项目管理业务流程**——5个业务场景，12个业务服务。关键决策是采用"问题"(Issue)而非"任务"(Task)作为核心领域概念，以及明确"分配问题给迭代"和"分配问题给项目成员"是独立业务服务而非属性编辑。

**培训业务流程**——3个业务场景，5个业务服务。这是最复杂的流程，票的生命周期涉及4个截止时间、3种处理策略、4种员工身份。取消票的流程更是分三种角色（员工、协调者、专员）分别处理。

回头看，这三个流程恰好构成了EAS的核心供需链条：客户合作创造需求，项目管理消耗资源，培训补充供给。泳道图中封装的子流程在服务蓝图中得到了完整展现。

几个深刻的体会：

第一，服务蓝图比单纯的流程图更适合业务分析。它区分了客户、前台、后台和支持者，让角色定位一目了然。市场人员看不见财务的参与、通知发送不属于独立业务服务——这些判断都得益于服务蓝图的层次结构。

第二，统一语言是在实践中逐步浮现的，不是预先定义的。每当我发现多个概念混用、歧义丛生时，就是统一语言需要建立的信号。关键是要"大声说出来"——在业务服务图可视化之后，对每一个可能产生歧义的概念及时消除分歧。

第三，全局分析阶段到业务服务这个粒度就可以结束了，避免分析瘫痪。但核心子领域的业务服务规约可以并行细化，作为领域建模阶段的输入。

接下来要进入架构映射阶段了。三大业务流程的业务场景和业务服务已经就位，下一步是确定限界上下文的划分。这将是最激动人心的部分——从问题空间到解决方案空间的跨越。
