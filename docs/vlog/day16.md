# Day 16 — 应用边界与最终限界上下文

**日期**：架构映射 Day 16
**作者**：张毅

---

今天是V型映射过程的最后一步：应用边界的识别。这一步从技术维度考量限界上下文，包括系统的质量属性、模块的复用性、对需求变化的应对和处理遗留系统的集成。

第一个要解决的是报表上下文的去留。之前在工作边界分析时，我考虑到报表上下文与其他限界上下文之间存在强依赖关系，无法支持并行开发，因而将报表功能分配给了其他限界上下文。如今通过技术分析得知，虽然依赖仍然存在，但该上下文更多地体现了"决策分析"的特定领域。客户希望统计报表能够准确及时地展现历史和当前的人才供需情况，统计报表功能直接影响了目标系统的愿景，是核心功能之一。最终，我决定保留该限界上下文，并将其更名为决策分析上下文。

第二个要解决的是通知上下文的定位。在考虑通知上下文的实现时，基于之前确定的系统上下文，EAS系统要与集团现有的OA系统集成。我们了解了OA系统公开的服务接口，发现这些接口已经提供了多种消息通知功能，包括站内消息、邮件通知和短消息通知，没有必要在EAS系统中重复开发通知功能。那么，通知上下文是否就没有存在的必要呢？一旦去掉通知上下文，与OA系统集成的功能又该放在哪里？领域驱动设计建议将这种与第三方服务集成的功能放在防腐层，可EAS系统中的多个限界上下文都需要调用该功能，会形成防腐层的重复建设。为了满足功能的复用性，可以为它单独创建一个限界上下文。为了说明其意图，将它更名为OA集成上下文。

至此，EAS系统的限界上下文划分工作全部完成。最终确定的12个限界上下文是：订单上下文、合同上下文、员工上下文、储备人才上下文、招聘上下文、项目上下文、培训上下文、决策分析上下文、文件共享上下文、OA集成上下文、组织上下文、认证上下文。其中前8个属于核心子领域，文件共享和OA集成属于支撑子领域，组织和认证属于通用子领域。

回顾这几天的推导过程，从业务主体的初步归类到最终12个限界上下文的确定，每一步调整都有据可依。这正是V型映射过程的价值——它不是一蹴而就的划分，而是一个逐步收敛的推导过程。明天将进入上下文映射环节，确定限界上下文之间的协作关系。
