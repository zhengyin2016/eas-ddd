# Day 3 — HR业务流程梳理

**日期**：Day 3
**参与者**：张毅（我）、李华（需求分析师）、王强（HR总监）、周涛（架构师，旁听）

---

今天终于进入实质性业务调研阶段了。李华约了HR总监王强做深度访谈，我和周涛一起参加。

说实话，之前我对HR模块的认知比较浅——觉得不就是增删改查员工信息嘛。今天跟王强聊完，才发现里面门道远比我想的多。

上午九点，李华准时拉着王强进了会议室。王强是个务实的人，不绕弯子，上来就在白板上画了他理解的HR全景图。他说HR的核心不是管人，是管"人与组织的关系"。这句话一下把讨论拉到了正确的高度。

我们花了整个上午梳理了五个核心流程：员工入职、员工离职/调岗、招聘管理、考勤管理、培训管理。每个流程李华都引导王强从"前台触点"和"后台支撑"两个维度展开。这是李华的拿手好戏——他总能把一个看似简单的操作拆解成前端交互和后端逻辑两层。

让我印象最深的是员工入职流程。表面上看就是"创建一个员工记录"，但王强说实际操作远不止于此。新员工入职时，必须关联到具体的部门和岗位，否则后续的考勤、薪资、项目分配全都会出问题。而且入职不是一锤子买卖——创建员工记录、分配工号、关联组织结构、创建系统账号、发送欢迎通知、分配导师、安排入职培训，这是一整条链路。工号的生成还有规则：年份加四位序号，每年重新计数。

调岗和离职的流程也很讲究。王强特别强调，离职必须完成所有项目交接才能走，不是提交申请就完事的。调岗则需要目标部门确认接收，否则可能出现"两头不管"的局面。他把员工状态拆成了六个：待入职、试用期、正式、调岗中、离职中、已离职。这个状态流转就是一个重要的生命周期，后面做领域建模时必须作为核心关注点。

下午我在白板上把员工入职的完整流程画了一遍。从HR创建员工开始，到分配工号、关联部门岗位、创建账号、通知员工、分配导师、安排入职培训，一气呵成。画完之后我退后两步看了看，这简直就是一张服务蓝图的雏形。

周涛在旁边一直在记笔记。画到一半的时候他突然抬头说："员工和组织的关系，是不是应该设计成一个聚合？"我赶紧拦住他："别急，现在还在全局分析阶段，先别急着设计模型。我们先把业务搞清楚，模型的事后面再说。"周涛不好意思地笑了笑，说习惯了，一看到业务概念就想往代码上靠。我理解他，但对团队来说，现在最重要的是建立对业务的共同理解，而不是急着设计方案。

王强还提到了一些容易忽略的细节。比如招聘管理——每个招聘需求必须关联到具体的部门和岗位，面试至少安排两轮，第一轮技术面、第二轮综合面。培训管理中，签到率低于百分之八十的培训需要重新安排。考勤管理中，迟到、早退、缺勤需要自动标记，请假必须走审批流程。

这些业务规则看似琐碎，但每一跳都是系统的约束条件。如果不在建模阶段把它们识别出来，后面开发时就会遗漏，然后就是返工。

关于绩效管理，王强主动说这次先不做。他说绩效涉及到考核指标、评分标准、申诉流程，太复杂了，建议放到二期。我同意。DDD强调的是边界划分，砍掉不确定的部分，集中精力做好确定的部分，这本身就是一种战略。

今天最大的收获，是我意识到"业务流程梳理"不是简单地记录CRUD操作，而是要理解业务背后的动机、规则和约束。服务蓝图是个好工具——前台触点描述了用户的直接体验，后台支撑描述了系统需要完成的工作，两者之间的映射关系就是我们需要建模的核心。

明天赵敏会来做PM业务流程的梳理。PM模块是EAS系统的另一个核心，涉及项目立项、人员分配、任务管理、进度跟踪。我预计复杂度不会比HR低。希望今天总结的方法论——前台触点加后台支撑的服务蓝图方法——明天还能用得上。

李华今天效率很高，五个流程的服务蓝图都整理出来了，晚上应该能出初稿。我让他明天早上发给我review，顺便把访谈记录也整理一下，作为需求文档的附件。

Day 3，进度正常。团队节奏在慢慢建立。
