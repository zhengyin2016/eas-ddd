# Day 3 — 核心业务流程浮现

今天开始进入业务需求分析阶段。前面完成了价值需求分析，搞清楚了 EAS 的利益相关者、系统愿景和范围。现在要让这些需求"动"起来——通过业务流程把静态的功能需求变成角色的协作过程。

核心发现：EAS 的业务愿景是"人力资源的供需平衡"，那么所有参与角色都可以归入两个阵营——需求方和供应方。站在这个角度去思考，协作关系一下子清晰了。

我们画了一张协作图（图20-5）。需求方包括市场部（提出客户需求）、项目管理部（提出项目用人需求）、人力资源部（提出招聘和培训需求）；供应方包括子公司（提供人力资源）、人力资源部（提供招聘供给）、培训专员（提供培训供给）。这张图不是当前状态，而是我们期望的未来状态——打破部门信息壁垒之后的协作方式。

基于这张协作图，我又画了核心流程的泳道图（图20-6）。从市场部接到客户需求开始，子公司评估资源，签订合同形成需求订单；项目管理部立项，分配项目成员和硬件资源，执行项目计划；人力资源部根据供需缺口启动招聘和培训。整个泳道图把核心流程的宏观运行过程展现出来了。

但是泳道图太宏观了，很多协作细节被封装在子流程里。比如项目管理流程和招聘流程，在泳道图里只是一个框。要真正理解问题空间，这些业务流程需要单独展开。

我决定用服务蓝图（Service Blueprint）来呈现具体的业务流程。服务蓝图有一个好处：它明确区分了客户、前台员工、后台员工和内部支持者的角色层次，比单纯的流程图更能体现"谁在什么层面做什么事"。

根据 EAS 的业务目标，我们确定了三个主要的业务流程：

1. **客户合作的业务流程**——面向市场人员
2. **项目管理流程**——面向项目管理人员
3. **培训流程**——面向培训专员

这三个流程覆盖了 EAS 的核心业务。今天先把框架搭起来，明天开始逐个深入梳理。先从客户合作开始，因为它是整个供需链条的起点——没有客户需求，就没有后续的一切。

今天的感悟：找到"供需关系"这个主线，就像找到了业务的脊梁骨。所有角色、所有流程都可以围绕这条主线展开。DDD 全局分析阶段的价值就在于此——不是急于建模，而是先把问题空间理清楚。
