# Day 8: 系统上下文定义

**日期**: 2026-05-24
**阶段**: Phase 2 - 架构映射
**参与者**: 张毅、王强、赵敏、孙磊、郑丽

## Vlog：绘制系统上下文图

今天我们正式进入架构映射阶段。第一件事就是搞清楚：EAS在整个企业IT环境中处于什么位置？

我画了个大圆圈代表EAS系统，然后问大家：EAS需要和哪些外部系统打交道？

王强第一个说：OA系统啊，员工入职、离职、组织架构变更，OA都有流程，EAS需要同步这些信息。

赵敏补充：还有薪资系统，HR算完考勤和绩效，需要推给薪资系统算工资。

孙磊说：财务系统也需要，合同签订后要触发开票、回款认领。

郑丽问了一个好问题：OA的这些功能——请假、审批、公文流转——我们要不要也做进去？

会议室安静了几秒。我说：这个问题问得好。但答案是：**不做**。

为什么？因为DDD的核心原则之一就是**明确系统边界**。

如果你什么都要做，最后就会变成一个什么都做、什么都做不好的巨石怪物。OA是公司已有的成熟系统，EAS应该通过接口集成，而不是重复造轮子。

这就是DDD的智慧：**不是所有业务都要进同一个限界上下文，甚至不是所有业务都要进同一个系统**。

我们最终确定的伴生系统：

| 伴生系统 | 集成方式 | 交互内容 |
|---------|---------|---------|
| OA系统 | REST API + 消息队列 | 组织架构变更、员工入职离职事件 |
| 薪资管理系统 | 数据同步/消息队列 | 考勤数据、绩效数据推送 |
| 财务系统 | REST API | 合同信息推送、开票申请、回款认领 |

郑丽又问：那OA和EAS谁是上游谁是下游？

好问题。从数据的"源头"来说，OA是上游——组织架构、员工信息都在OA里先产生。但从业务流程来说，EAS发起"分配员工到项目"的调用时，EAS是主动调用方，OA是服务提供方。

这就是为什么我们说上下游关系是**场景相关的**，不是绝对的。

## 今天的产出

我们画出了C4 Level 1的系统上下文图——虽然只是草图，但大家心里有数了：EAS不是孤岛，它需要和其他系统协作，但协作不等于融合。
