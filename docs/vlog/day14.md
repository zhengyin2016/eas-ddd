# Day 14: 架构映射阶段评审

**日期**: 2026-05-30
**阶段**: Phase 2 - 架构映射
**参与者**: 张毅、王强、赵敏、孙磊、郑丽

## Vlog：7天工作总结

今天做架构映射阶段评审。总结一下7天的成果：

### Day 8: 系统上下文定义

我们画了C4 Level 1系统上下文图，明确了EAS与伴生系统（OA、薪资、财务）的关系。更重要的是，我们**否决了把OA功能纳入EAS的想法**——明确系统边界是DDD的核心原则。

### Day 9: V型映射与限界上下文识别

通过V型映射过程，将32个业务服务组织成6个限界上下文。讨论了HR是否应该拆分，最终决定**保持单一上下文，内部按聚合划分**——这是30天项目的实用平衡。

### Day 10: 菱形对称架构

为每个上下文设计了菱形对称架构——北向网关(OHS)、领域层、南向网关(ACL)。确定了技术栈：Spring Boot 3.2 + MyBatis + MySQL。

### Day 11: 上下文映射与服务契约

发现**PM是协作中心**——它需要从HR获取人员信息，从CRM获取合同信息。定义了各上下文的REST API契约。

### Day 12: 系统分层架构

画了C4 Level 2容器图。决定**不加API Gateway**，用Spring Boot的DispatcherServlet就够了——30天项目能省就省。

### Day 13: 统一语言词典

建立了统一语言词典，定义了50+个术语。强调了统一语言是**限界上下文级别**的——比如"项目"在PM和CRM里含义不同。

## 评审结果

评审通过。大家对架构方向达成一致，可以进入Phase 3建模与实现。

王强说HR这块准备开干了。
赵敏说PM的聚合设计比较有挑战。
孙磊说CRM的商机流转状态机得仔细设计。

郑丽说基础层（组织、认证、通知）她先搭好架子，让三个核心上下文能跑起来。

## 三个架构决策记录(ADR)

评审结束后，我把四个关键决策写成了ADR：

1. ADR-001: 选择MyBatis而非JPA
2. ADR-002: 采用菱形对称架构
3. ADR-003: HR模块保持单一限界上下文
4. ADR-004: PM上下文作为协作中心
