# Day 2 — 商业模式画布

今天一整天都在做商业模式画布，过程比预想的曲折。

上午从客户细分开始。团队很快列出了市场部、项目管理部、人力资源部这几个显而易见的角色——它们确实是 EAS 的直接用户。市场部管理客户关系，项目管理部跟踪项目进度和人员分配，人力资源部负责招聘和培训。但讨论到价值主张时，我们发现一个问题：这些部门的管理者确实需要 EAS 来提高工作效率，但真正关心"人力资源供需平衡"这个核心目标的人是谁？

答案是集团管理层——具体来说，是集团的决策者。这个角色一开始居然被我们遗漏了。

下午重新梳理了商业模式画布。在客户细分中补上了集团管理层之后，整个画面清晰了很多。决策者的痛点在于：无法快速直观地了解当前人力资源的供给和需求状况。项目在抢人，招聘跟不上，培训周期长，这些信息散落在各个部门和系统里，决策者看到的是一堆碎片，不是一幅完整的图景。

关于核心资源，讨论也有分歧。有人说是技术平台，有人说是数据。但回到本质：我们是软件集团，核心生产力就是从事软件开发的人力资源。技术可以换，平台可以搭，但人是不可替代的。所以核心资源就是人力资源本身。

价值主张也做了调整。对市场部而言，价值在于提高客户关系管理效率；对项目管理部而言，是项目过程管理的规范化和透明化；对人力资源部而言，是招聘计划的前瞻性——不再是被动招人，而是根据供需预测提前规划。

渠道方面倒没有争议。EAS 是集团内部系统，由集团下达行政命令在内部职能部门推广，不需要像商业产品那样寻找获客渠道。收益来源也明确了：一方面是成本控制和削减，减少人力资源错配带来的浪费；另一方面是为集团的软件交付业务提供更稳定的人力支撑。

重要合作主要和人力资源招聘与培训有关——集团有合作的培训公司，有招聘网站的简历资源，还有软件学院作为人才储备。成本结构则主要是开发这套系统本身的投入。

今天最重要的收获不是画布本身，而是讨论过程中暴露出来的认知盲区。如果不是系统地走完商业模式画布，我们很可能一开始就忽略了决策者这个最关键的利益相关者。

明天开始梳理业务流程，从当前状态出发，看看现有的业务是怎么运转的。
