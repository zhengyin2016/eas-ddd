# Day 4 — 客户合作：从流程到统一语言

今天深入梳理客户合作的业务流程，收获很大——不只是识别出了业务服务，更重要的是经历了一次关键的统一语言建立过程。

先看服务蓝图。市场人员发起"创建市场需求"的服务请求，子公司作为合同乙方评估需求、签订合同，然后市场人员基于合同创建需求订单。这里有几个关键角色定位：子公司是前台员工，需要和市场人员直接交互；财务是后台员工，核算行为发生在子公司与财务之间，市场人员看不见；市场人员是服务蓝图的客户角色，不参与合同签订，只关心资源是否满足。

根据服务蓝图，客户合作流程分为4个业务场景：

1. **市场需求管理**：创建市场需求、评估市场需求、查询市场需求。其中"查询市场需求"虽然没出现在服务蓝图中，但如果不提供这个功能，子公司就无法获得指定的市场需求来完成评估，所以必须单独定义。

2. **简历管理**：导出员工简历。看起来简单，但实际上需要子公司操作人员与系统进行交互才能完成导出。

3. **合同管理**：创建合同、添加附加合同、指定合同承担者、归档合同、查询合同。注意，合同签订在线下进行，EAS 只负责维护合同信息并归档。

4. **需求订单管理**：创建需求订单、跟踪需求订单、查询需求订单。

画完业务服务图，我发现几个领域概念的定义非常模糊：合同、市场需求、客户需求、需求订单——它们之间的关系是什么？有什么区别？

和市场人员交流后，情况更复杂了。他们有时说"录入市场需求"，有时又说"跟踪市场需求订单"，这些概念在他们的日常交流中是混用的。他们提到客户需求需要指定"承担者"，但市场需求不需要。对于合同，大家一致认为这是法律概念，是甲乙方签订的合作协议。

**这是建立统一语言的关键时刻。**

我们决定引入"订单"(Order)概念。借鉴电商系统，订单描述的是市场部与客户达成的合作意向。订单不是需求。一个订单可以包含多个客户需求，相当于订单项。比如同一个客户提出三条需求：需要5名高级Java程序员和10名中级程序员；需要8名初级.NET程序员；需要开发一个OA系统——这三条客户需求组成一个订单。

引入订单后，市场需求和客户需求的区别也清楚了：市场需求是售前人员了解到的需求，未经过评估，公司不知道能否满足；客户需求是经过评估后的需求，有明确的承担者。市场需求评估后转化为订单中的客户需求。

合同保留，但和订单本质不同。一个订单的每个客户需求可以由不同子公司承担，但合同只能有一个甲方和一个乙方。订单没有法律条款。未签合同的内容确实很大部分来自订单，但只是商务合作内容的一部分。

今天的教训：统一语言不是一开始就能定义清楚的，而是在梳理业务服务的过程中逐步浮现的。当发现多个概念混用、歧义丛生时，必须停下来，大声说出来，及时消除分歧。
