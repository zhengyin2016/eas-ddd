# Day 4 — 赵敏的秩序感：项目管理业务流程梳理

**日期**：Day 4
**访谈对象**：赵敏（PM总监）
**参与者**：张毅、李华、吴斌、赵敏

---

今天轮到赵敏了。

赵敏是我们公司PMO的总监，做了十几年项目管理，从PMP到PRINCE2各种认证拿了个遍。她最大的特点就是对"规范性"有着近乎执念的追求——任何事情都要有流程、有标准、有据可查。

说实话，刚开始做DDD实战的时候，我有点担心赵敏会不适应。毕竟领域驱动设计强调的是"探索"和"涌现"，而赵敏习惯的是"定义"和"控制"。但事实证明我多虑了，赵敏对业务的深刻理解让她在领域建模方面有天然的优势。

上午九点，我们准时开始。李华主导访谈，我在旁边画流程图，吴斌负责记录。

赵敏一上来就给我画了一张项目生命周期状态图：

```
筹备 → 已立项 → 进行中 → 已结项
              ↓         ↑
             暂停 ──────┘
```

她说："张毅，你看，项目状态有严格的转换规则。筹备状态只能转到已立项，已立项可以转到进行中或暂停，进行中可以转到暂停或已结项，暂停只能转回进行中。不能跳过状态，也不能回退。"

我追问："那暂停状态有特殊规则吗？"

赵敏说："暂停期间，项目成员理论上应该被释放，但实际操作中我们往往是'冻结'状态——人还在项目上，但不计入工时。这个细节很重要，因为它影响HR系统的员工可用性计算。"

这个细节确实关键。我把它记了下来。

接着我们梳理了四个核心业务流程：项目立项、人员分配、进度跟踪、项目结项。每个流程赵敏都讲得非常细致，甚至连审批节点的超时处理都想到了。

人员分配这个环节引发了最多的讨论。赵敏描述了一个让我非常兴奋的跨领域交互场景：PM分配人员时，需要从HR上下文查询员工的可用性和技能信息。这简直就是教科书级的上下文映射案例——PM上下文需要依赖HR上下文提供的数据，但不应该直接操作HR的数据。

吴斌在这个环节问了一个好问题："赵敏姐，项目中的Issue和项目是什么关系？每个项目都必须有Issue吗？"

赵敏笑了笑："Issue是项目中的问题跟踪，类似于轻量级的问题管理。不是每个项目都必须有——小项目可能从头到尾都不需要Issue。但大项目、特别是涉及多部门协作的项目，Issue管理是必不可少的。Issue有自己的严重程度和优先级体系，跟任务是两个不同的东西。任务是有计划的，Issue是计划外冒出来的问题。"

这个区分很清晰。我在白板上写下了"Task = Planned, Issue = Unplanned"。

下午的讨论中，我发现了一个赵敏没有明确说出来但实际存在的业务规则：一个员工同一时间只能全职参与一个项目，但可以兼职参与多个项目。这个"全职/兼职"的区分对后续的领域建模至关重要——它意味着ProjectMembership这个实体需要一个commitmentType属性（FULL_TIME / PART_TIME），而且HR上下文在计算员工可用性时也需要考虑这个属性。

我把这个发现跟赵敏确认了一下，她说对，这个规则一直存在，但之前的系统从来没有正确处理过。"经常出现一个人同时被全职分配到两个项目的情况，两边项目经理都以为这个人是自己专属的，结果工时冲突，成本核算也一塌糊涂。"

这让我更加确信，DDD的价值不仅在于技术架构，更在于把那些"大家都知道但没人说清楚"的业务规则显式化、代码化。

今天最大的收获是明确了PM上下文与HR上下文的边界。PM负责"需求"（我需要什么样的人），HR负责"供给"（谁可用、有什么技能）。人员分配的本质是PM上下文向HR上下文发起一个"人员查询请求"，拿到结果后由PM做决策。这个交互模式应该在后续的上下文映射中用Anti-Corruption Layer来保护。

明天轮到CRM业务流程梳理了。陈浩那个人虽然大大咧咧的，但销售出身的人对业务的理解往往比我们想象的深。

**关键发现**：
1. 项目状态有严格的转换规则，不能跳过或回退
2. PM与HR之间存在清晰的跨上下文交互：人员查询与分配
3. 员工的全职/兼职参与模式是一个核心业务规则
4. Task（计划内）与 Issue（计划外）是两个不同的概念
5. 项目暂停期间的人员"冻结"状态需要特殊处理
