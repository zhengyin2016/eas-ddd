## 2. 特殊限界上下文的内部架构

### 2.1 决策分析上下文——弱化的菱形对称架构

决策分析上下文的领域逻辑主要为统计分析，受技术决策的影响，通常可以直接针对数据进行操作，无须建立领域模型，形成弱化的菱形对称架构。内部只包含北向网关与南向网关。南向网关是一个薄薄的数据访问层，从数据库获得的统计分析数据会直接转换为消息契约模型。

```
┌─────────────────────────────────────┐
│       决策分析上下文（弱化菱形）       │
│                                     │
│  ┌──── 北向网关 ──────────────────┐ │
│  │  remote / local / appservice   │ │
│  └────────────────┬───────────────┘ │
│                   │                 │
│  ┌────────────────┴───────────────┐ │
│  │  （无领域模型层）               │ │
│  └────────────────┬───────────────┘ │
│                   │                 │
│  ┌────────────────┴───────────────┐ │
│  │  南向网关（薄数据访问层）        │ │
│  │  统计数据 → 直接转为消息契约    │ │
│  └────────────────────────────────┘ │
└─────────────────────────────────────┘
```

### 2.2 OA集成上下文——由防腐层发展而来

OA集成上下文是一个由防腐层发展起来的限界上下文。它与其他限界上下文的协作采用了发布者/订阅者模式，内部又需要调用OA系统的服务接口。它的领域层只包含了组装消息内容的领域模型。

- 事件订阅者 = 北向网关的远程服务
- 事件处理器 = 北向网关的应用服务
- 事件发布者 = 南向网关（分为端口与适配器）

### 2.3 文件共享上下文——基础层的库

文件共享上下文实现为基础层的库（非服务）。它封装的领域逻辑除了上传文件与下载文件的领域行为，还规定了属于不同类别的文件存放在文件服务器的不同位置。文件传输的实现属于南向网关适配器的内容。

以归档合同为例，合同上下文调用文件共享上下文的FileAppService（对应图20-49）。

## 3. 菱形对称架构代码模型

对应图20-50。所有限界上下文都采用了菱形对称架构规定的标准代码模型，只是根据具体情况作了少量调整。

```
eas-{context}/
│
├── message/                          # 发布语言 (PL)
│   └── {消息契约模型类}               # 事件、命令、查询参数、返回值
│
├── northbound/                       # 开放主机服务 (OHS)
│   ├── remote/                       # 远程服务
│   │   ├── resource/                 # REST资源
│   │   ├── controller/               # 控制器
│   │   ├── provider/                 # 服务提供者
│   │   └── subscriber/               # 事件订阅者（OA集成上下文使用）
│   ├── local/                        # 本地服务
│   └── appservice/                   # 应用服务
│
├── domain/                           # 领域层（按聚合划分命名空间）
│   └── {aggregate}/
│       ├── {Entity}
│       ├── {ValueObject}
│       ├── {DomainService}
│       └── {Repository接口}
│
└── southbound/                       # 防腐层 (ACL)
    ├── port/                         # 端口
    │   ├── repository/               # 资源库接口
    │   └── client/                   # 客户端接口
    └── adapter/                      # 适配器
        ├── repository/               # 资源库实现
        └── client/                   # 客户端实现
```

### 各命名空间说明

| 目录 | 角色 | 说明 |
|------|------|------|
| `message/` | 发布语言 (PL) | 消息契约模型，包括事件、命令、查询参数、返回值类型 |
| `northbound/remote/` | 开放主机服务-远程 | REST资源、控制器、服务提供者、事件订阅者 |
| `northbound/local/` | 开放主机服务-本地 | 进程内本地服务 |
| `northbound/appservice/` | 开放主机服务-应用服务 | 应用服务，编排领域逻辑，公开服务契约 |
| `domain/` | 领域层 | 按聚合划分命名空间，包含实体、值对象、领域服务 |
| `southbound/port/` | 防腐层-端口 | 资源库接口和客户端接口（抽象） |
| `southbound/adapter/` | 防腐层-适配器 | 资源库实现和客户端实现（具体） |

### 设计约束

- 所有服务契约的方法参数和返回值都是消息契约的一部分
- 不能直接将领域模型暴露在外
- 各个限界上下文在系统分层架构中所处的层次，通过包的命名空间清晰地呈现
- 系统分层架构属于架构的逻辑视图，不需要体现通信边界的物理差异
