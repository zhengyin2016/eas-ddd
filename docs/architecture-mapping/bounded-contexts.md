# 限界上下文识别 (Bounded Contexts)

**编号**: EAS-ARCH-002
**版本**: 1.0
**日期**: 2026-05-25
**作者**: 张毅

## 概述

本文档通过V型映射过程，将32个业务服务映射为6个限界上下文，并定义每个上下文的职责边界。

## V型映射过程

```
                    业务服务层 (32个服务)
                           │
                           ▼ V型映射
                    ┌──────┴──────┐
                    │  分组决策   │
                    │  聚内聚     │
                    │  斩耦合     │
                    └──────┬──────┘
                           ▼
                    限界上下文层 (6个上下文)
```

### 映射原则

1. **单一职责原则**：每个限界上下文只负责一个明确的业务领域
2. **高内聚低耦合**：相关的服务放在一起，交互频繁的服务分开
3. **团队边界对齐**：上下文边界尽量与团队结构匹配
4. **技术独立性**：每个上下文可独立部署、使用不同技术栈

## 技术决策记录

### 决策001：HR模块保持单一限界上下文

**背景**: HR模块内部可以拆分为员工、招聘、培训、考勤等多个独立上下文。

**决策**: 为简化项目管理，本次实施将HR统一为一个限界上下文，内部通过聚合划分。

**理由**:
- 30天项目时间有限，拆分会增加集成成本
- 三个后端开发，每人负责一个核心上下文
- 未来可以根据需要拆分

**影响**: HR上下文内部需要通过模块化保持聚合间低耦合
