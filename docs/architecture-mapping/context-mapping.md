## 1. 服务序列图推导上下文映射

### 1.1 创建市场需求

创建市场需求业务服务属于订单上下文，它拥有的领域知识已经足以满足该业务服务的需求，无须求助于其他的限界上下文。因此在本业务服务中，没有上下文映射。

虽然创建市场需求没有多个限界上下文参与协作，但为其绘制服务序列图仍有必要，因为可以通过它驱动出创建市场需求的服务契约。

对应图20-42。

---

### 1.2 归档合同

归档合同业务服务属于合同上下文，具备合同相关的领域知识，但不具备上传文件的业务能力，需要求助于文件共享上下文。

文件共享上下文作为支撑子领域的限界上下文，主要提供了文件上传与下载的功能。它具有的领域知识还包括针对不同类型的文档维护了服务器文件存储的路径映射，故而参与协作的是北向网关的应用服务。

**上下文映射模式**：客户方/供应方（Customer/Supplier）

- 合同上下文 = 客户方（下游）
- 文件共享上下文 = 供应方（上游）

对应图20-43、图20-44。

---

### 1.3 添加项目成员

添加项目成员业务服务属于项目上下文，拥有与项目相关的所有领域知识和业务能力，但并不具备发送通知的业务能力，也不知道该如何将当前项目的信息添加到员工的项目经验中，因此需要分别求助于OA集成上下文和员工上下文。

**与OA集成上下文的协作**：

OA集成上下文要实现的功能都与通知有关。无论是短信通知、邮件通知还是站内通知，都没有副作用，且允许以异步形式调用，适合使用事件的调用机制。

- **上下文映射模式**：发布者/订阅者（Publisher/Subscriber）
- 项目管理者在添加了项目成员之后，会向事件总线发布 TeamMemberAdded 应用事件
- OA集成上下文的事件订阅者订阅该事件

选择该模式既可以解除OA集成上下文与大多数限界上下文之间的耦合，又能较好地保证EAS系统的响应速度，减轻主应用服务器的压力。不足是需要增加一台部署消息队列的服务器，并在一定程度上增加了架构的复杂度。

**与员工上下文的协作**：

- **上下文映射模式**：客户方/供应方（Customer/Supplier）
- EmployeeAppService 追加项目经验

对应图20-45、图20-46。

## 3. 映射模式说明

| 模式 | 方向 | 说明 |
|------|------|------|
| 客户方/供应方 | 合同上下文 → 文件共享上下文 | 合同上下文请求文件上传/下载，文件共享上下文提供FileAppService |
| 客户方/供应方 | 项目上下文 → 员工上下文 | 项目上下文请求追加项目经验，员工上下文提供EmployeeAppService |
| 发布者/订阅者 | 项目上下文 → OA集成上下文 | 项目上下文发布TeamMemberAdded事件，OA集成上下文订阅并转发OA系统 |

## 4. 服务契约设计

服务序列图驱动我们获得了消息定义，由此可以驱动出服务契约（对应表20-5）。

### 通信边界

在确定EAS的限界上下文时，并未明确指出限界上下文的通信边界。在无法给出必须跨进程通信的证据之前，应优先考虑进程内通信。

EAS系统作为企业内部系统，对并发访问与低延迟的要求并不高。可用性固然重要，但EAS系统毕竟不是"生死攸关"的一线生产系统，即使短时间出现故障，也不会给企业带来致命的打击或难以估量的损失。

**通信边界决策**：
- 所有限界上下文优先采用**进程内通信**
- 唯一例外：**OA集成上下文**采用**进程间通信**（需跨进程调用OA系统）
- OA集成上下文通过事件总线实现分布式通信

### 核心服务契约

| 供应方上下文 | 服务类 | 服务方法 | 映射模式 | 说明 |
|------------|--------|---------|---------|------|
| 文件共享上下文 | FileAppService | upload() / download() | 客户方/供应方 | 文件上传与下载，维护路径映射 |
| 员工上下文 | EmployeeAppService | appendProjectExperience() | 客户方/供应方 | 追加员工项目经验 |
| 项目上下文 | TeamAppService | publish TeamMemberAdded | 发布者/订阅者 | 发布应用事件（支持分布式通信） |
| OA集成上下文 | 事件订阅者 | subscribe TeamMemberAdded | 发布者/订阅者 | 订阅事件，调用OA系统发送通知 |

### 服务契约约束

- 所有服务契约的方法参数和返回值都是消息契约的一部分
- 满足菱形对称架构要求：不能直接将领域模型暴露在外
- 采用客户方/供应方模式的供应方通过应用服务对外公开服务契约
- 采用发布者/订阅者模式的发布方通过应用服务发布应用事件
- TeamMemberAdded事件属于应用事件，需要支持分布式通信

## 服务契约（对应书中表20-5）

| 上下文映射模式 | 服务方法 | 参数 | 返回值 |
|---------------|---------|------|--------|
| 客户方/供应方 | FileAppService.upload(contractId, file) | 合同ID, 文件 | 文件路径 |
| 客户方/供应方 | EmployeeAppService.appendProjectExperience(employeeId, projectExperience) | 员工ID, 项目经验 | void |
| 发布者/订阅者 | TeamAppService.publishTeamMemberAdded(event) | TeamMemberAdded事件 | void |
| UI→应用服务 | NominationAppService.nominate(request) | NominationRequest | NominationResponse |
| UI→应用服务 | ContractAppService.archiveContract(contractId, file) | 合同ID, 文件 | void |
| UI→应用服务 | TeamAppService.addTeamMember(projectId, employeeId, role) | 项目ID, 员工ID, 角色 | void |

通信边界说明：
- 除OA集成上下文外，所有限界上下文采用进程内通信
- OA集成上下文定义为进程间通信（跨进程调用OA系统）
- 客户方/供应方模式：供应方通过应用服务公开服务契约
- 发布者/订阅者模式：通过应用服务发布应用事件
