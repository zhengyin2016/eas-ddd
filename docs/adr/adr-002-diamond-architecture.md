# ADR-002: 采用菱形对称架构

**状态**: 已接受
**日期**: 2026-05-30
**决策者**: 张毅
**影响范围**: 全项目架构

## 上下文

需要定义每个限界上下文的内部架构模式。候选方案：
- 传统三层架构（Controller-Service-DAO）
- DDD分层架构（用户界面-应用-领域-基础设施）
- 菱形对称架构（OHS-领域-ACL）

考虑因素：
- 限界上下文之间的交互
- 外部依赖隔离
- 领域模型保护

## 决策

采用**菱形对称架构**模式，每个限界上下文分为四层：

```
    北向网关 (OHS) - 开放主机服务
         应用层
         领域层
    南向网关 (ACL) - 防腐层
```

## 理由

1. **OHS明确对外契约**：北向网关定义稳定的API，保护领域模型
2. **ACL隔离外部变化**：南向网关防腐，外部变化不影响领域
3. **对称性**：OHS和ACL对称，架构清晰
4. **上下文集成友好**：每个上下文通过OHS暴露，通过ACL调用

## 后果

### 正面

- 限界上下文间松耦合
- 外部依赖变化时影响范围小
- 领域模型纯粹，不被外部污染

### 负面

- 多一层抽象，代码量增加
- 学习成本，团队需要理解OHS/ACL概念

### 缓解措施

- 提供脚手架代码模板
- 文档和培训

## 架构示意

```
        ▲ OHS (北向网关)
       ╱ ╲ REST API
      ╱   ╲
     ╱     ╲ 领域层
    ╱       ╲
   ▼ ACL (南向网关)
```

**OHS职责**：
- 定义REST API
- 请求参数验证
- DTO与领域对象转换
- 事件发布

**ACL职责**：
- 调用外部上下文API
- 数据格式转换
- 异常处理与重试
- 降级策略

## 替代方案

### 传统三层架构

**优点**：团队熟悉，简单直接

**缺点**：
- 缺少明确的防腐层
- 外部依赖容易侵入领域
- 限界上下文边界不清晰

**拒绝理由**：多上下文项目，需要更强的边界保护
