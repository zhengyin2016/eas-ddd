# ADR-004: PM上下文作为协作中心

**状态**: 已接受
**日期**: 2026-05-30
**决策者**: 张毅
**影响范围**: PM限界上下文、系统集成架构

## 上下文

通过上下文映射分析发现：

PM上下文需要与多个其他上下文交互：
- 从HR获取员工信息和可用性
- 从CRM获取客户和合同信息
- 从组织获取部门架构
- 向HR发送人员释放事件

PM处于"协作中心"的位置。

## 决策

**接受PM作为协作中心的架构定位**，并做好相应设计准备。

## 理由

1. **业务现实**：项目管理天然需要协调人员、客户、资源
2. **数据汇聚点**：PM是业务信息的汇聚地
3. **事件枢纽**：接收CRM事件，向HR发送事件

## 后果

### 正面

- 架构清晰，PM作为业务编排中心
- 符合业务实际

### 负面

- PM可能成为性能瓶颈
- PM的可用性要求更高

### 缓解措施

- PM的ACL调用加缓存
- 非核心路径支持降级
- PM服务做好监控和弹性设计

## PM的集成点

| 调用方向 | 目标 | 内容 |
|---------|------|------|
| ACL调用 | HR上下文 | 查询员工信息、可用员工 |
| ACL调用 | CRM上下文 | 查询客户、合同信息 |
| ACL调用 | 组织上下文 | 查询部门、岗位 |
| 事件消费 | CRM上下文 | 合同签订事件 |
| 事件发布 | HR上下文 | 人员释放事件 |

## 替代方案

### 引入工作流引擎

**优点**：
- 工作流可视化
- 流程可配置

**缺点**：
- 增加复杂度
- 30天项目学习成本高
- 过度设计

**拒绝理由**：30天项目，业务流程固定，不需要工作流引擎

### 事件驱动架构

**优点**：
- 松耦合
- 可扩展

**缺点**：
- 调试困难
- 最终一致性的业务复杂度

**拒绝理由**：PM需要实时查询人员可用性，同步调用更合适
