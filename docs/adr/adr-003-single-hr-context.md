# ADR-003: HR模块保持单一限界上下文

**状态**: 已接受
**日期**: 2026-05-30
**决策者**: 张毅
**影响范围**: HR限界上下文

## 上下文

HR模块包含12个业务服务：员工管理、招聘、面试、考勤、培训、技能档案等。

分析发现HR内部可以拆分为多个独立上下文：
- 员工上下文：员工全生命周期
- 招聘上下文：招聘需求、面试
- 培训上下文：培训计划、报名
- 考勤上下文：打卡、请假统计

## 决策

HR模块**保持单一限界上下文**，内部通过聚合划分边界，不拆分为多个上下文。

## 理由

1. **时间限制**：30天项目，拆分会增加集成成本
2. **人员配置**：三个后端开发，每人负责一个核心上下文
3. **数据一致性**：员工信息在HR内部高度关联
4. **内部模块化**：通过聚合划分仍然可以保持内部高内聚

## 后果

### 正面

- 开发效率高，减少跨上下文集成
- 数据事务简单，同一数据库
- HR开发人员工作量可控

### 负面

- HR上下文职责较重
- 如果HR需要独立团队，后续可能需要拆分

### 缓解措施

- HR内部按聚合严格划分模块
- 保持聚合间低耦合
- 未来可以根据需要拆分

## 内部聚合划分

虽然不拆分上下文，但内部按聚合划分：

| 聚合 | 职责 |
|-----|------|
| 员工聚合 | 员工基本信息、状态 |
| 招聘需求聚合 | 招聘需求审批 |
| 面试聚合 | 面试安排与评估 |
| 考勤聚合 | 考勤记录 |
| 培训聚合 | 培训计划与报名 |
| 技能聚合 | 技能档案 |

## 替代方案

### 拆分为多个限界上下文

**优点**：
- 职责更单一
- 可独立部署和扩展

**缺点**：
- 需要更多开发人员
- 跨上下文集成复杂
- 30天时间不够

**拒绝理由**：时间与人员限制，本次实施不拆分
