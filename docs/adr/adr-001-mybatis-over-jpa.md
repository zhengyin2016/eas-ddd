# ADR-001: 选择MyBatis而非JPA

**状态**: 已接受
**日期**: 2026-05-30
**决策者**: 张毅
**影响范围**: 全项目

## 上下文

EAS项目需要选择ORM框架来处理数据库访问。候选方案主要是MyBatis和JPA（Hibernate）。

考虑因素：
- 团队技术背景：团队成员更熟悉SQL
- 查询复杂度：EAS有较多报表查询需求
- 性能要求：核心业务需要高性能查询
- 开发效率：30天项目时间紧张

## 决策

选择**MyBatis**作为ORM框架，不使用JPA/Hibernate。

## 理由

1. **SQL可控性**：MyBatis保留SQL的完全控制权，利于性能优化
2. **复杂查询**：报表查询、多表关联用SQL更直观
3. **团队熟悉度**：团队成员SQL经验丰富，JPA学习成本高
4. **调试友好**：SQL直接写在mapper里，问题排查更容易
5. **Schema演进**：MyBatis对数据库变更更宽容

## 后果

### 正面

- SQL性能优化更直接
- 复杂查询实现更简单
- 团队上手更快

### 负面

- 需要手写更多SQL代码
- 缺少JPA的领域模型映射能力
- Entity和PO对象需要手动转换

### 缓解措施

- 使用MyBatis-Plus减少CRUD代码
- 定义统一的Converter处理对象转换
- 共享Mapper接口模板

## 替代方案

### JPA/Hibernate

**优点**：
- 标准JPA注解，领域模型映射自然
- 自动生成SQL，开发效率高
- 二级缓存支持

**缺点**：
- 复杂查询需要Criteria API，可读性差
- N+1查询问题常见
- 团队不熟悉，学习成本高

**拒绝理由**：30天项目，报表查询多，SQL可控性更重要
